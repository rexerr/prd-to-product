## Backlog

- **Validation finding: skill chain doesn't auto-compose; mental model mismatch + misleading repo name.** First observation: 2026-05-11 Squirreled validation. User invoked `/context-engineering` with a working brief, expecting the chain (prd-creator → context-engineering → design-system-bootstrap) to run as one orchestrated flow. The current architecture has three separate skills that hand off manually — composition is the user's responsibility. Result: user skipped prd-creator entirely. context-engineering gracefully accepted the brief in place of a PRD, but the resulting PRD will be thinner than a prd-creator interview would produce. **Two connected issues:** (1) sophisticated user mental model assumes one command runs the chain; (2) the repo name `prd-to-product` implies you start with a PRD when the actual starting point is a rough idea / brief — a more accurate name is `idea-to-product`. The rename reinforces the architectural fix: `/idea-to-product` is the natural name for a meta-skill that orchestrates. **Three fix candidates:** (A) build `/idea-to-product` meta-skill that orchestrates the chain; (B) smart routing in context-engineering cluster 0 — detect brief-vs-PRD and offer to call prd-creator first; (C) cheapest — rename the repo to `idea-to-product` + better cluster-0 question + README. Recommend C as first move, escalate to B if C doesn't solve, escalate to A if multi-step orchestration earns its keep. Rename cost: ~30 minutes (GitHub auto-redirects, ~15 in-repo find/replace, no symlink breakage). **Defer all of this until Squirreled validation completes** — major finding worth its own iteration session.
- **Validation finding: prd-creator's "don't pre-fill, ask anyway" rule overcorrects on thorough briefs.** First observation: 2026-05-11 Squirreled validation. User has a thorough brief covering modes, scope, and explicit V1/V2 calls; the skill nonetheless asked them to list V1 capabilities cold ("Give me a bullet list of capabilities you'd need to check off before you'd say 'this is shippable'"). User reasonably pushed back: "doesn't my brief tell you what it is? why do I need to?" The intake.md "ask anyway" discipline exists to prevent silent absorption (model rubber-stamps the brief without depth), but currently collapses two distinct behaviors: ❌ silent absorption (bad) and ✅ proposal-with-consent (good). The model can do "draft from the brief, show me, let me edit" but defaults to asking cold because intake.md treats both as the same forbidden pattern. **Fix:** modify intake.md cluster-question wording — "when the brief covers the cluster topic, draft an answer and present for edits; only ask cold when the brief is thin." Applies to every cluster, not just scope. Likely the highest-impact intake.md edit of all findings so far. Defer until validation completes.
- **Validation finding: prd-creator hallucinates temporal context.** First observation: 2026-05-11 Squirreled validation. Skill said "the brief was written days ago and your framing may have sharpened" — but the brief was written today (the "Source" line in the brief explicitly states the date). The model defaulted to "days ago" as a generic phrasing without grounding in any actual timestamp it could verify. Pattern: the skill makes plausible-sounding factual claims about source-material provenance without evidence. The interview intent was fine ("get current phrasing, not paraphrase") but the false premise undermines trust. Fix: add an instruction to `skills/prd-creator/generator/intake.md` — "do not speculate about when source material was created or how recent it is. Reference source material without temporal claims." Likely same fix needed in `context-engineering`. Defer until validation completes.
- **Validation finding: skills should auto-detect input, not require explicit attachment.** First observation: 2026-05-11 Squirreled validation. Current behavior: user must explicitly attach the brief with `@/path/to/brief.md` syntax when invoking `/prd-creator` or `/context-engineering`. User expectation: open the project folder, run `/prd-creator`, and have the skill find the brief in the working directory automatically. If found → confirm with user ("I see `docs/brief.md` — is this what you want to work from?"). If multiple candidates → list and ask which to use. If none → ask user whether to paste content, type from scratch, or proceed without source material. **Where to look:** project root, `docs/`, common filenames (`BRIEF.md`, `brief.md`, `*-brief.md`, `IDEA.md`). Connects to the chain-orchestration finding above (same root cause: skills require explicit input plumbing the user shouldn't have to think about). Fix surface: cluster 0 in both `prd-creator` and `context-engineering`. Likely also applies to `design-system-bootstrap` for brand assets. Defer until validation completes.
- **Validation finding: prd-creator leaks internal vocabulary to the user.** First observation: 2026-05-11 Squirreled validation. The skill exposes implementation detail in its user-facing prompts — "Cluster 0," "0a / 0b" sub-question IDs, "as source material," "extract cold," likely also "D-NNN" decision IDs as the interview proceeds. The user sees how the sausage is made instead of a conversation. Underlying principle: **internal vocabulary stays internal; user-facing text reads as natural language.** Specific incident (cluster 0 split into Y/N triage instead of the prescribed single open question) is the visible symptom of the broader leak. Likely fix surface: `skills/prd-creator/generator/intake.md` (rewrite all cluster intros to not name the cluster; rewrite branching as conversational prompts not lettered sub-questions), possibly `skills/prd-creator/SKILL.md` and `skills/prd-creator/principles.md` (add the principle as a stated rule). Scope unknown until full validation completes; may extend to `context-engineering` and `design-system-bootstrap` if they exhibit the same leak. **Defer the fix until the Squirreled validation completes** — don't change the skill mid-experiment. Aggregate with any other onboarding friction the validation surfaces.
