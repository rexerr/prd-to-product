# Agent-process & context-harness upgrades — portable handoff

A self-contained writeup of the reasoning, problems, and solutions from a Séance (iOS app)
session that hardened the project's **agent-process layer** — the slash commands, CLAUDE.md
rules, settings, and memory hygiene that govern how an AI coding agent works in the repo.

It is written to be **lifted out of Séance and handed to two audiences:**

- **The `context-engineering` skill maintainer** — bake these into the scaffolder so every new
  project starts with them (see **Part A**).
- **Any existing already-scaffolded project** — a retrofit checklist + a drop-in prompt the
  project's own agent can run to self-improve (see **Part B**).

Throughout, *universal patterns* are kept separate from *project-specific parameters* so nothing
Séance-specific (e.g. "confirm on a physical iPhone") gets globalized blindly. Anywhere you see
`{{PLACEHOLDER}}`, that's a per-project value the harness fills at scaffold time.

---

## 0. The triggering question — global vs project-specific session commands

**Question:** I had `/session-start` as a *global* skill. Should the harness instead generate a
project-specific `/session-start` and `/end-session` per project?

**Verdict: project-specific, generated by the harness from a universal template. Keep a global
copy only as a thin fallback.**

Reasoning:

- A useful session command references things **only the project knows**: its roadmap path, its
  decisions-log layout, its retro directory, *how "done" is proven* (device confirmation vs. unit
  tests vs. a browser preview), its commit-URL format, its branch policy, and *what needs human
  sign-off*. A global command can't know these, so it either hardcodes wrong assumptions or stays
  so generic it enforces nothing.
- The **pattern** is universal (orient by reading only what isn't already loaded; close out by
  restate → update docs → retro → bundle → push → hand off). The **parameters** are per-project.
  That split is exactly what a scaffolder is for: own the template, emit a filled-in copy.
- A checked-in `.claude/commands/*.md` also **binds every agent on the repo** (Codex, other tools),
  is versioned, and travels with the project — a global skill does none of that.

So: the harness should emit `/session-start` and `/end-session` wired to the project's real values,
and you can keep a generic global pair as a default for throwaway repos that were never scaffolded.

## 1. Universal patterns discovered (the *why* behind each)

Each pattern is stated as **problem → solution → how to generalize**. These are the reusable ideas;
Parts A and B turn them into concrete harness changes and a retrofit checklist.

### 1.1 Don't read what's already in context
- **Problem:** The old `/session-start` re-read `CLAUDE.md` + `AGENTS.md` — but those **auto-load**
  at session start. ~1/3 of what the command read (~175 of ~505 lines) was pure waste, and it ran
  at full weight even for a trivial one-off.
- **Solution:** The command explicitly does **not** re-read auto-loaded files, and **scales heavy
  reads to the session**: always read the latest retro (cheap, carries the handoff); read the
  roadmap + active-decisions only when actually resuming feature work.
- **Generalize:** Any orientation step should read *only what isn't already in context*, and read
  the minimum that lets you act. "Read everything every time" is a false sense of safety that costs
  tokens and latency.

### 1.2 Memory ↔ repo boundary, and one source of truth
- **Problem:** A preference lived in the agent's private memory ("build only, the human is the
  eyeballs — don't drive a Simulator screenshot tool") while the checked-in CLAUDE.md said the
  **opposite** ("use the self-check tool before asking"). The two silently contradicted for weeks.
  Other memories merely *duplicated* repo docs (Figma source-of-truth, skills list) — drift waiting
  to happen.
- **Solution:** A clear test for where a fact belongs:
  - **Repo** (CLAUDE.md / docs) if it must bind *every* agent (incl. Codex/other tools) **or** it
    refines/contradicts an existing repo rule.
  - **Memory** if it's personal or cross-project *working style* that isn't natural repo content
    (who the user is, how they like to be worked with across projects).
  - When a rule lands in the auto-loaded CLAUDE.md, **delete the duplicate memory** — one source of
    truth. Memory and repo drifting apart is the failure mode.
- **Generalize:** Memory is for the agent's relationship with the *person*; the repo is for rules
  that govern the *work*. Never let the same rule live authoritatively in both.

### 1.3 The autonomy charter — "run to done," with a *narrow* human gate
- **Problem:** The agent stopped to ask permission at every step ("want me to commit?"), producing
  exactly the back-and-forth and token waste the user hates — while *also* not clearly protecting
  the decisions that genuinely need a human.
- **Solution:** Write an explicit **autonomy charter** into CLAUDE.md:
  - **Non-visual / reversible work** (logic, docs, tests, config): run the whole loop in one turn —
    edit → build → test → retro → commit → push — then report. Don't ask permission to *finish*.
  - **Still gated on the human — never self-clear:** UI/visual changes (visual confirmation),
    product / architecture / scope decisions, and anything **irreversible or outward-facing**
    (public release, deleting the user's work, force-push).
  - **Big forks** (new phase, architecture change) → use plan mode: propose, get approval, execute.
- **Generalize:** "More agentic without losing human-in-the-loop" is a *boundary* problem, not a
  *throttle* problem. Don't slow everything down uniformly — write down the few things that need a
  human and run hard on everything else. The boundary is the user's call; everything inside it is
  the agent's.

### 1.4 Permissions allowlist — stop prompting for safe reads
- **Problem:** Read-only commands (`git status`, `grep`, build, test) triggered approval prompts —
  round-trips with zero risk.
- **Solution:** A `.claude/settings.json` `permissions.allow` list for read-only / non-mutating
  commands. Writes, commits, and pushes are **deliberately left off** — they run under the autonomy
  charter and any commit-blocking hooks.
- **Generalize:** Pre-approve what can't change product state; gate what can. (There's also a
  `fewer-permission-prompts` skill that derives this from transcripts — the harness can *seed* a
  sensible list at creation so it's good from day one.)

### 1.5 Retro discipline — in-commit, before-commit, timestamped, conditional
- **Problem 1 (double-push):** The agent committed the code, *then* wrote the retro as a second
  commit — the exact waste the retro is supposed to avoid.
- **Problem 2 (same-day ambiguity):** Retro filenames are date-only (`YYYY-MM-DD-topic.md`), but
  several sessions happen per day. A retro said a setting "was not done this session," which got
  misread as "not done at all" — when an *earlier same-day* commit had already done it. Nothing
  ordered the work within the day.
- **Solution:**
  - Write the retro **before** the code commit and **bundle both into one commit/push**.
  - **Timestamp the retro H1** and note which session of the day it is
    (`# Retro — YYYY-MM-DD HH:MM TZ — topic   (Nth session of the day)`); get the time with
    `date "+%Y-%m-%d %H:%M %Z"`.
  - Only write a retro for **non-trivial** sessions (skip one-line tweaks) — avoid retro-spam.
- **Generalize:** Any artifact that records "what happened this session" needs *intra-day ordering*
  and should ship *with* the work it describes, not as a trailing commit.

### 1.6 Bookend commands — `/session-start` + `/end-session`, project-wired
- **Problem:** There was an opening ritual but no closing one, so the close-out (update roadmap +
  decisions, write retro, bundle, push, hand off) depended on the agent *remembering* it — and it
  fumbled the ordering.
- **Solution:** A matching `/end-session` that codifies the close-out and **mechanically enforces**
  retro-in-the-same-commit and the human commit gate (if a UI/visual change isn't confirmed, it
  refuses to commit and says what to look at).
- **Generalize:** Rituals you rely on the agent to *remember* should be *commands*, not prose hopes.
  Bookend every session: orient in, hand off out.

### 1.7 Hooks turn prose into enforcement
- **Pattern already in use:** load-bearing rules that can be mechanically checked become hooks
  (e.g. a PreToolUse hook that blocks staging secrets/signing material on `git add`; a PostToolUse
  formatter). Prose says "don't commit secrets"; the hook *guarantees* it.
- **Generalize:** For each non-negotiable rule, ask "can a hook enforce this?" If yes, emit a hook —
  don't rely on the model reading the rule. (Don't over-hook; reserve for the genuinely load-bearing.)

### 1.8 Surface conflicts; don't silently duplicate
- **Pattern:** The memory-vs-CLAUDE.md contradiction survived because nothing ever reconciled the
  two. A periodic reconcile pass (there's a `consolidate-memory` skill for the memory side) catches
  drift. AGENTS.md being a *thin pointer* to CLAUDE.md (not a copy) is the right anti-duplication move.
- **Generalize:** When the same rule could live in two places, pick one authoritative home and
  cross-reference; schedule a reconcile so copies can't drift undetected.

## 2. Part A — changes to bake into the `context-engineering` skill

The scaffolder should **interview for a few more things**, then **emit a few more artifacts**.

### 2.1 Add to the scaffold interview
Collect (with sensible defaults so it stays fast):

1. **Verification method — "how is *done* proven here?"** e.g. `xcodebuild build` + on-device visual
   confirmation (iOS); `npm test` + a browser preview (web); CLI run; etc. → feeds the verification
   rule, the commit gate, and `/end-session`.
2. **Human-gate boundary (autonomy charter).** "What must *you* sign off on vs. what should the agent
   run to completion?" Offer the default split (UI/visual + product/architecture/scope +
   irreversible/outward-facing are gated; logic/docs/tests/config run to done). → emits the Autonomy
   section + the commit gate.
3. **Docs topology.** Paths for roadmap, decisions log (+ an "active decisions" mirror if used), and
   retros dir. → wires both session commands.
4. **Commit conventions.** Remote/commit-URL format, co-author trailer, branch policy
   (e.g. "direct on main, no branches" vs. PR flow). → wires `/end-session`.
5. **Mechanically-enforceable non-negotiables.** Secrets/keys to block on `git add`, formatter to run
   on edit, anything else a hook can guarantee. → emits hooks.

### 2.2 Emit these artifacts (new or upgraded)
1. **`.claude/commands/session-start.md`** — the lean orient template (§Appendix A), filled with the
   project's docs paths. Explicitly does *not* re-read auto-loaded `CLAUDE.md`/`AGENTS.md`.
2. **`.claude/commands/end-session.md`** — the close-out template (§Appendix B), wired to the project's
   docs, commit convention, and commit gate.
3. **`.claude/settings.json` → `permissions.allow`** — seed a read-only allowlist (§Appendix C),
   including the project's build/test command and read-only MCP tools; keep writes/commits/pushes off.
4. **CLAUDE.md → "Autonomy — run to done, then report"** section (§Appendix D).
5. **CLAUDE.md → a short "Memory vs. repo" hygiene note** (§Appendix E) so agents file facts in the
   right place and delete duplicated memories.
6. **`docs/retros/README.md`** — the retro convention: write *before* the code commit, bundle into one
   commit, **timestamped H1 + session-of-day**, conditional on non-triviality (§Appendix F).
7. **Hooks** for the non-negotiables collected above (the skill already does secrets/format — keep it).

### 2.3 One meta-principle to add to the skill's own guidance
> **Prefer commands and hooks over prose for anything you rely on happening every time.** Prose in
> CLAUDE.md is read *probabilistically*; a command is invoked deterministically and a hook executes
> unconditionally. Reserve prose for judgment; encode rituals and guarantees.

## 3. Part B — retrofit checklist for an existing (already-scaffolded) project

Hand this to a project's own agent. It's a self-audit that applies the patterns above without a
re-scaffold.

**Drop-in prompt:**

> Audit and upgrade this project's agent-process layer using these patterns. Do the read-only audit
> first, show me the findings, then apply the fixes you're confident in (docs/config only) under the
> autonomy charter; ask me only about the human-gate boundary.
>
> 1. **Session commands.** Do `.claude/commands/session-start.md` and `end-session.md` exist? If a
>    start command re-reads `CLAUDE.md`/`AGENTS.md`, trim that (they auto-load). If there's no
>    `/end-session`, create one wired to our roadmap/decisions/retros paths, our commit convention,
>    and our commit gate.
> 2. **Memory vs. repo.** List every agent-memory file. For each, decide: does it bind every agent or
>    refine a repo rule (→ promote to CLAUDE.md/docs and delete the memory), does it merely duplicate
>    a repo doc (→ delete), is it stale (→ fix), or is it genuinely personal/cross-project (→ keep)?
>    Flag any memory that **contradicts** CLAUDE.md — reconcile in the repo's favor where it's a work
>    rule.
> 3. **Autonomy charter.** Is there an explicit "run to done vs. human-gated" boundary in CLAUDE.md?
>    If not, propose one (default: logic/docs/tests/config run to completion; UI/visual,
>    product/architecture/scope, and irreversible/outward-facing actions are gated). Ask me to ratify
>    the boundary.
> 4. **Permissions allowlist.** Add a `.claude/settings.json` `permissions.allow` for read-only
>    commands (git status/diff/log, grep/find/ls, our build + test commands, read-only MCP tools).
>    Keep writes/commits/pushes off it.
> 5. **Retro convention.** Update `docs/retros/README.md`: retro written *before* the code commit and
>    bundled into one commit; **timestamped H1 + session-of-day**; only for non-trivial sessions.
> 6. **Hooks.** Is each non-negotiable rule (no secrets committed, formatting, etc.) actually enforced
>    by a hook, or just stated in prose? Convert the load-bearing ones to hooks.
> 7. Bundle all doc/config changes into one commit with a clear message; show me the diff before push
>    if anything touches a gated area.

## 4. Appendix — parameterized templates

Replace `{{...}}` at scaffold/retrofit time. Defaults shown in comments.

### Appendix A — `/session-start`

```markdown
Orient me to resume. Read only what isn't already in context.

**Do NOT re-read CLAUDE.md or AGENTS.md** — they auto-load at session start and are already in
context. Opening them is wasted tokens.

**Always read** (cheap, and this is the actual resume signal):
- The most recent file in `{{RETROS_DIR}}` — sort by filename date; if several share a date, pick the
  latest timestamp in the H1. Its "Next session" section is the handoff.

**Read only if resuming build/feature work** (skip for a trivial one-off):
- `{{ROADMAP_PATH}}` — current phase + next uncompleted item.
- `{{DECISIONS_ACTIVE_PATH}}` — binding decisions not visible from code alone.

Then give me: one sentence on where we left off (from the retro), and the next uncompleted
{{ROADMAP_PATH}} item (only if you read it). Don't write code or propose solutions yet. Just orient.
```

### Appendix B — `/end-session`

```markdown
Close out the session cleanly. Follow CLAUDE.md "Autonomy" and the retro convention.

1. **Restate** in one short block: what was done, what was *verified* ({{VERIFY_METHOD}}), what remains.
2. **Update the docs the work changed:** tick `{{ROADMAP_PATH}}` checkboxes / mark a phase done when
   its "done-when" is met; log any binding decision in `{{DECISIONS_PATH}}` (+ mirror to
   `{{DECISIONS_ACTIVE_PATH}}` if it imposes a constraint not visible from code).
3. **Retro — only if non-trivial.** `date "+%Y-%m-%d %H:%M %Z"`; write
   `{{RETROS_DIR}}/YYYY-MM-DD-topic.md` with a timestamped H1 + which session of the day. Be honest
   about misses, deviations, and what verification did and didn't cover.
4. **Commit + push as ONE commit** — stage code + retro + doc updates together; never a separate
   second push for the retro. Paste the commit URL (`{{COMMIT_URL_FMT}}`).
   - **Commit gate exception:** if the change touches {{GATED_SURFACE}} and {{HUMAN}} hasn't confirmed
     it, do NOT commit. Stop, say exactly what to check, commit only after sign-off.
5. **Hand off:** list anything still gated on {{HUMAN}} and the single next thing to pick up.
```

### Appendix C — `.claude/settings.json` allowlist (merge into existing settings)

```json
{
  "permissions": {
    "allow": [
      "Bash(git status:*)", "Bash(git diff:*)", "Bash(git log:*)", "Bash(git show:*)",
      "Bash(git branch:*)", "Bash(git rev-parse:*)",
      "Bash(grep:*)", "Bash(rg:*)", "Bash(find:*)", "Bash(ls:*)", "Bash(cat:*)", "Bash(date:*)",
      "Bash({{BUILD_CMD}}:*)", "Bash({{TEST_CMD}}:*)"
      /* + read-only MCP tools, e.g. "mcp__Figma__get_metadata" */
    ]
  }
}
/* Writes/commits/pushes are intentionally NOT here — they run under the autonomy charter
   and any commit-blocking hooks. */
```

### Appendix D — CLAUDE.md "Autonomy" section

```markdown
### Autonomy — run to done, then report

Default to finishing the work, not asking permission to finish it. Concentrate the human's
involvement on the calls that actually need them.

- **Non-visual work** (logic, docs, tests, config): run the whole loop in one turn —
  edit → {{BUILD_CMD}} → {{TEST_CMD}} (when logic is touched) → retro (if warranted, before the
  commit) → commit → push — then report with the commit URL. Don't stop to ask "want me to commit?"
- **Still gated on {{HUMAN}} — never self-clear:** {{GATED_SURFACE}} changes (visual/behavioral
  confirmation); product / architecture / scope decisions; anything irreversible or outward-facing
  (public release, deleting their work, force-push). Surface or hand off; don't proceed.
- **Big forks** (new phase, architecture change): use plan mode — propose, get approval, then execute.
- Hard scope limits still bind; exceeding them is itself a "stop and confirm."
```

### Appendix E — CLAUDE.md "Memory vs. repo" hygiene note

```markdown
### Where facts live — memory vs. repo

- **Repo** (this file / docs): any rule that must bind every agent (incl. other tools) or that
  refines/contradicts a rule here. Repo rules are the source of truth.
- **Memory:** personal or cross-project working style only (who the user is, how they like to work).
- When a rule lands here, **delete the duplicate memory** — one source of truth. If a memory ever
  contradicts this file on a *work* rule, this file wins; reconcile and delete the memory.
```

### Appendix F — `docs/retros/README.md` convention

```markdown
Write the retro **before** the code commit and bundle both into one commit/push — never push the
code, then push the retro separately. Only for non-trivial sessions. Name files
`YYYY-MM-DD-topic.md`.

**Timestamp the H1.** Filenames are date-only but several sessions run per day; put the wall-clock
time + which session of the day in the title (`date "+%Y-%m-%d %H:%M %Z"`) so same-day retros are
orderable — otherwise "not done this session" gets misread as "not done at all" when an earlier
same-day commit already did it.

Template H1: `# Retro — YYYY-MM-DD HH:MM TZ — [topic]   (Nth session of the day)`
```

## 5. Honest caveats (don't globalize blindly)

- **Verification is the most project-specific axis.** Séance's gate is "live render on a physical
  iPhone" — most projects' is tests + a preview. The *pattern* ("prove done, don't claim it; gate the
  un-provable on a human") generalizes; the *method* must be re-collected per project.
- **The autonomy boundary is the user's call, every time.** The default split is a starting point, not
  a law — the harness should *ask*, not assume.
- **Don't over-hook or over-command.** Reserve hooks for load-bearing guarantees and commands for
  rituals you rely on; everything else stays prose so the agent keeps judgment.
- **Allowlist scope is a trust decision.** The read-only list above is safe; resist the urge to add
  anything that mutates state "for convenience."

## 6. This repo's reconciliations (added 2026-06-08, not from Séance)

Two places where this brief, applied verbatim, would conflict with `prd-to-product`'s existing
decisions. The `BACKLOG.md` entry carries the resolved plan; recorded here so the source and the
divergence sit together.

- **Memory model (Appendix E vs. `principles.md:19–25`).** Séance frames the cut as *personal vs.
  work*. This repo already framed it as *agent-discovered vs. user-decided*. For a user who moves
  between Claude Code, Codex, and Cursor, the operative cut is neither — it is **survives-a-tool-switch
  vs. doesn't**: Claude's auto-memory is machine-local and Claude-only (Codex/Cursor can't read it),
  so anything load-bearing *or* any cross-tool working preference must live in the repo
  (`AGENTS.md`/docs); auto-memory is Claude-local scratch only. Reconcile to that one definition
  before shipping Appendix E.
- **Autonomy charter (Appendix D vs. the scope-gate).** This repo's rules say "if the plan touches
  >2 files, wait for confirmation," which directly contradicts "run the whole loop in one turn."
  Resolution: scope limits are the **outer gate**; "run to done" governs only what is inside them.
  That precedence must be the load-bearing line, not a footnote.
