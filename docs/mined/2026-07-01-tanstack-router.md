### RT-3 — The freshness system, whole — and its measured limits · code-grounded · → brownfield-drift row

`library_version` pins make staleness **arithmetic**: 29 skills pinned `1.166.2`, one at `1.161.4`, package now `1.171.14` — the fleet lags ~5 minors and the pin exposes exactly which skill fell furthest. `sources:` manifests name what each skill distills (repo-qualified doc paths; degrading gracefully to `packages/vue-router/src` where no docs exist yet). `check-skills.yml` (**`intent-workflow-version: 4`** — the refresh loop we recorded as unclosed in intent is closed here): PRs touching skills run `intent validate`; releases + pushes touching `docs/**` or `packages/*/src/**` run `intent stale --github-review`, opening/updating **one rolling review PR** (`skills/review-${VERSION}`) — drift lands as a single reviewable unit, not N nags. And yet: the artifacts rotted anyway. Shipped skills are **fresher than their planning artifacts** (a Zod v3/v4 nuance, a new CRITICAL auth mistake, and a whole security skill exist in skills but not in any `_artifacts` inventory) — maintenance happens at the skill; artifacts are birth certificates, not living registries. Root artifact copies went stale one rename behind their re-homed per-package successors.

### RT-4 — Seq-1 project-setup inputs · code-grounded · → project-setup handoff row + supply-chain row

- **Supply-chain defense at the package-manager layer, not CI** (`pnpm-workspace.yaml:2-8,105-145`): `minimumReleaseAge: 1440` (no dep younger than 24h installs — publish-hijack damper), `trustPolicy: 'no-downgrade'` with commented exceptions, `blockExoticSubdeps: true`, and an `allowBuilds` postinstall allowlist where **every entry carries a rationale comment** ("'@prisma/engines': false # only required for auth e2e examples"). Reframes the banked "no vuln/secret scanning even in mature foundations": the defense exists at install time; first-party code scanning is still absent.
- **Auto-repair CI class** (`autofix.yml:25-32`): formatting + regenerated configs are committed back onto the PR ('ci: apply automated fixes'), not failed — a third gate class beyond pass/fail and advisory.
- **CI as a target-name vocabulary**: the whole PR gate is `nx affected --targets=test:eslint,test:unit,test:e2e,test:types,test:build,build` — orthogonal gates expressed as a contract every package implements, with compute sized to the diff.
- **Per-PR installable preview packages** (`pkg-pr-new publish` + example templates) — reviewers test the artifact, not the diff.
- **Security as e2e regression suites**: literal attack URLs with fix rationale in comments (`open-redirect-prevention.spec.ts:12-13` — `/%0d/google.com/` resolving protocol-relative) plus whole CSP and import-protection suites, and a "normal navigation still works" over-fix guard.
- **e2e-utils architecture**: harness = cross-cutting invariants only (port arbitration via files, hermetic fake external host, a **console-error-strict page fixture** failing any test on stray browser errors, HMR file editor, dev-server warmup, history-mode path abstraction); each suite is a real tiny app. Test-matrix-as-data: packages declare `playwrightModes` metadata; a local Nx plugin generates the vite-ssr/spa/prerender × rsbuild target matrix.
- **Config as derived artifact**: labeler config generated from the package tree (auto-refreshed by autofix); the TS-version support matrix maintained by one script fanning out; renovate deliberately **excludes the deps that ARE the test matrix** (typescript, react, node) — automation stops where the dep is a fixture.
- **Examples-as-fixtures**: 111 examples are workspace packages; freshness is a hook inside `changeset:version` (rewrite every example dep to the new version at release time); `catalog:` + blanket `workspace:*` overrides keep example package.jsons copy-paste-valid for users while building from source; affected examples build in PR CI.
- **Release**: changesets + branch-derived dist-tags; the GitHub-release script deletes the tag it created on failure (compensating rollback — second sighting after CE).

### RT-6 — Docs-craft · code-grounded · → docs-craft row

- **`ref:` + `replace:` thin pointers** — 24 solid docs are 9-line files: a pointer to the react source + a declarative rename map applied at render time (`docs/start/framework/solid/guide/authentication-overview.md`). Dedup-by-transform: divergence is impossible for ref'd pages and *visible* (a real file appearing where a ref was IS the divergence event). The closest external analog to our dry-run-substitution machinery found anywhere.
- **Two IA strategies chosen per product by divergence level**: router docs = shared files with `<!-- ::start:framework -->` inline variant blocks + per-section framework overlay slots in the nav manifest; start docs = fully forked per-framework trees. No global rule — divergence level decides.
- **The plan-rot cautionary tale**: three agent-campaign planning docs at repo root (issue-mined priority backlog → implementation plan with embedded reusable agent prompt + status board → verification ledger). The campaign *worked* (all 15 guides shipped; testing found and fixed real issues) — then the docs rotted four ways in place: stale paths to a restructured tree, two internal status tables diverged, a ledger claiming "**Perfect Success Rate**: 100% (6/6)" while admitting "6/15 guides tested", and no retirement. Index-rot beats content-rot (the drafts README row outlived its shipped-and-deleted draft). Direct support for our plan-retirement ritual and doc-routing rule.
- **Comment-out-future-links discipline**: "Common next steps" links are committed commented-out until the target guide exists, enforced by the plan checklist — write-the-link-before-the-target without rot (12 of 25 guides carry the blocks).
- **How-to taxonomy**: task-focused / self-contained / copy-paste-ready how-tos vs conceptual guides, with a mandated per-guide skeleton — and the whole 25-guide corpus absent from the site nav manifest (complete on disk, un-navigable).
