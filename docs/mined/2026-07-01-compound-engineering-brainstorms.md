## Findings by landing surface (namespaced CEB-N)

### Pressure-test ticket (the biggest payoff — verified mechanisms)

- **CEB-1 — Contract tests assert skill *prose*. · verified shipped.** `tests/pipeline-review-contract.test.ts` / `review-skill-contract.test.ts` / `skills/unified-plan-artifact-contract.test.ts` are string-assertions that a skill's wording holds — e.g. "ce:work checklist says 'Testing addressed', not 'Tests pass'" (`2026-03-29-testing-addressed-gate-requirements.md:44-49`). This is a *shipped* answer to the pressure-test's core question "does the wording hold" — pair it with ASCE-5's measured router benchmark (does the wording change *behavior*) and the ticket has both halves: assert-the-string (cheap, deterministic) + measure-the-behavior (expensive, statistical).
- **CEB-2 — Observable-behavior-first framing bar, with a testable prose spec.** Findings must "lead with the observable behavior … not the code structure," explain *why the fix works*, cite an in-repo pattern, in a 2-4 sentence budget, with a worked weak/strong exemplar pair (`2026-04-17-ce-review-interactive-judgment-requirements.md:77-87`); ported once via a *shared subagent template*, not per-persona (`2026-04-18-...:140`). A concrete, gradeable prose bar — exactly what the pressure-test measures.
- **CEB-3 — LLM-judge design (from the shipped ce-optimize + doc-review).** Judge only after free gates pass (order: degenerate gates → hard metrics → judge-costs-money → keep/revert, `iterative-optimization-loop-requirements.md:496-501`); stratified sampling for cost (~30 samples, $30-90/100-experiments, `:453-473`); **fixed `sample_seed` across runs** so you compare the same items (`:502-506`); **anchor-sample judge-drift detection** — a fixed set with known scores in every batch; if anchors shift >0.5, recalibrate (`:885-889`); **rubric-change invalidates prior comparisons** → re-baseline (`:890-893`). Plus per-severity confidence gates (P0 0.50 / P1 0.60 / P2 0.65 / P3 0.75, `doc-review-...:44-48`) and strawman-aware classification ("a do-nothing option is not a real alternative," `:33`). The judge-drift-anchor and seed-fixing are grader-calibration mechanics beyond what the ticket currently specs.

### loop-graduation + chain-auto-compose (ce-optimize — a shipped optimization loop)

- **CEB-4 — ce-optimize is a production metric-driven loop. · verified shipped.** Nearly every principle those rows track, in one shipped skill: **immutable measurement harness** (`scope.mutable`/`scope.immutable`; the judge prompt is immutable — the agent can't game the metric, `SKILL.md:99,281`); **three-tier metrics** (degenerate gates → LLM-judge → diagnostics-not-gated); **measurement-first HARD gate** (refuses the loop until the harness validates + baseline recorded); **multiple stop/kill conditions** (target / max-iter 100 / max-hours 8 / plateau-10 / manual); **dependency-approval boundary** (pre-approve expected, defer surprises to a batch ask so unattended runs don't block); **Codex 3-failure kill-switch** + `CODEX_SANDBOX` recursion guard; **noise-floor gate** (don't accept improvements within the noise). The load-bearing lesson: *"Hard scalar metrics are often the wrong optimization target … quality requires judgment"* — a proxy metric (fewer singletons) looked good while the judge caught the real quality drop (`iterative-optimization-loop-requirements.md:80,330`).
- **CEB-5 — Durable-log write-then-verify. · verified shipped.** The first run lost 90 minutes because results lived only in context; the fix: mandatory disk checkpoints CP-0..CP-5, each a *write-then-verify* cycle, with the rule *"If you produce a results table in the conversation without writing those results to disk first, you have a bug"* (`SKILL.md:78`; `iterative-optimization-loop-requirements.md:961-963`). A sharper instance of lfg's "residuals-durable-then-exit" and our own checkpoint discipline. Experiment log is structured YAML (hypothesis/changes/before-after/outcome/learnings) read before proposing the next step; reverted experiments logged so failed approaches aren't retried.
- **CEB-6 — Two "learning-across-runs" storage stances.** Stateless-via-external-authority (a skill detects its own prior runs from its GitHub reply-comments, zero extra API calls, GitHub is source of truth — `cross-invocation-cluster-analysis-requirements.md:19-22`) vs persistent-local (read auto-memory). Both read-only toward their source. The stateless-external stance is the cheaper, drift-free option for safe unattended learning.

### llm-council (persona architecture)

- **CEB-7 — Persona activation + method.** Two-tier: always-on personas (coherence, feasibility) + conditional ones activated by **content judgment, not keyword match** (`plan-review-personas-requirements.md:16-26`; `cli-readiness-...:19`) — "cost proportional to document complexity." Each persona carries a named *review method*, not just a domain: premise-challenge ("what's the 10-star version? the narrowest wedge that proves demand?"), dimensional rating ("rate 0-10 and describe what 10 looks like"), scope-guardian's "what already exists?" codebase check. Dispatched in parallel → deduped → synthesized; overlap handled at synthesis, not prevented.
- **CEB-8 — Confidence discipline.** Cross-persona agreement *boosts confidence of findings that already passed the gate* (+0.10 cap) — it does not create findings (`doc-review-...:49`). Per-severity confidence gates (not a flat threshold). Advisory tier is gate-exempt — a safety valve for on-record-but-not-escalated.
- **CEB-9 — Interactive-judgment UX + the anti-ritual test.** The named failure: bucket-level review "degrades into rubber-stamping." Fix: a four-option routing menu upfront (engage-each / trust-agent-per-item / batch-defer-to-tracker / report-only), per-finding walk-through with a "Finding 3 of 8 (P1):" position indicator and a mid-flow bail-out, bulk-action plan-preview + Proceed/Cancel, deterministic conservative tie-break (Skip > Defer > Apply). The **anti-ritual success criterion**: prove the intervention *shifts engagement upward*, not just renames the rubber-stamp (`review-interactive-judgment-requirements.md:103`). Directly relevant to whether llm-council's synthesis is real or ceremonial.

### furnace-plan

- **CEB-10 — Plan-doc anatomy.** Sections: Implementation Units / Requirements Trace / Files / Test Scenarios / Verification (`ce-plan-rewrite-requirements.md:80`); decisions-sequencing-risks, *not* implementation code; **portable — no executor litter** (no TodoWrite/`/ce:work` choreography/git recipes in the artifact); verification expressed as *expected outcomes, not commands with predicted output*; task sizing = logical units/atomic commits, not 2-5-minute steps.
- **CEB-11 — Blocker-classification gate.** A product blocker (changes behavior/scope/success criteria) must pause the plan; a technical/research question may be *reclassified and absorbed*. On a real blocker: surface it, let the user convert it to an explicit assumption/decision, or route back to brainstorm (`ce-plan-rewrite-requirements.md:24-26`).
- **CEB-12 — Approach-altitude ("plan for a plan").** Answer one level up first — *how* the deliverable will be made — before committing to it; **"the hold is the point of the feature"** (the user shouldn't have to fight the tool to prevent auto-execution). High-precision/low-recall trigger to avoid the "new-hammer failure mode" (every turn opening with "want me to plan the approach first?"); offer only when method-uncertainty AND cost-of-wrong are both high; two-stage grounding (cheap heuristic decides whether to offer → light recon only after accept → deep ingestion deferred to execution). The real boundary is **code vs. knowledge-work**, not plan vs. execute.
- **CEB-13 — Auto-deepen.** The plan evaluates and strengthens its own weak sections *without asking* (agent-decides, user-informed) because "the user can't evaluate this better than the agent"; self-gates on scope + risk-weighted confidence; reports what it's strengthening (transparency without a decision); folded in as a *phase*, not a sub-skill invocation (eliminates a boundary).

### prd-creator

- **CEB-14 — Scope-surfacing craft.** A **Stated / Inferred / Out-of-scope** synthesis presented *before* the doc is written — "forces honesty about how much was assumed vs. agreed" (`surface-scope-earlier-requirements.md:36-39,117`); **soft-cut on circularity** (same item revised twice), not iteration count; inline focused Q&A ~3 as a guideline not a hard cap; **open-prose corrections, not a menu** (a menu "would leak the agent's framing of valid corrections"); falsifiable success criteria on the change itself ("if the downstream effects don't materialize, the diagnosis was wrong — a real signal, not a partial win"). In headless mode, omit the Inferred list — propagating un-validated inferences as authoritative is unsafe.

### brainstorm skill

- **CEB-15 — ce-ideate structure.** Generate-many → **adversarially reject weak ones (the core quality mechanism)** → detail only survivors; survivor record = description/rationale/downsides/confidence/complexity + a one-line-per-rejected rejection summary; **durable-artifact-compounds** ("discarding ideation results is anti-compounding" — rejected ideas prevent re-exploring dead ends); orchestrator owns ranking, sub-agents emit-only in a standardized format; distinct-perspective framings (friction / unmet-need / inversion / assumption-breaking / leverage / extreme-case); present-before-write; resume artifacts <30 days old.
- **CEB-16 — Issue-grounded ideation.** Ideate from *tracker patterns, not individual issues* — "25 duplicate bugs about the same failure mode is a signal about collaboration reliability, not 25 separate problems"; theme-level output (what the pattern is / why it matters / what it signals); cost-tiered reading (titles+labels for all, full bodies for 2-3 representatives per cluster); hybrid frames (derive from clusters, pad with defaults when <4 clusters); <5 issues → fall back to default ideation with a note.

### skill-craft (feeds Seq-2 consolidation)

- **CEB-17 — Persona catalog as a maintained reference file** (activation + severity guidance per entry); **distill a heavy standalone reviewer into a lightweight in-pipeline persona** while the standalone stays for direct invocation; persona *self-scopes* after dispatch, orchestrator only decides *whether* to dispatch.
- **CEB-18 — Menu discipline** (4-option cap owned as a real constraint; labels self-contained, third-person, front-load the distinguishing word to survive truncation; tool pre-loaded once at mode start, text fallback only on explicit miss). *Dedup-adjacent with ASCE + our AskUserQuestion tracking — a 3rd sighting.*
- **CEB-19 — "Replace, don't layer"** (a new pipeline fully replaces the old mode, no back-compat shim, "simpler mental model, one behavior"); separate *how to engage* from *what to do next*; **generalize the taxonomy rather than add a mode flag** (avoid special-case accretion); the **instruction-forcefulness lesson** — "the skill had the right machinery but the instructions didn't forcefully guide the agent to use it … instructions need to actively detect the right path and guide toward it" (matches our every-line-changes-behavior rule).
- **CEB-20 — Defer-to-the-artifact-itself** (append findings to a `## Deferred / Open Questions` section *in the document*, not an external tracker — "deferred findings stay attached to the document they came from"); no-silent-failure-on-defer (retry / fall back to next sink / convert-to-skip-with-error-recorded); tracker detection is reasoning-based, not an enumerated file checklist.
- **CEB-21 — Multi-round memory + suppression** (carry a cumulative decision primer; personas must not re-raise a finding rejected in a prior round; fixed findings self-suppress because their evidence is gone; synthesis flags "fix did not land" when a prior-applied finding re-surfaces); institutional-memory persona runs advisory and does not compete for user attention; compressed search-seed (not full doc) handed to the memory persona.

### config-scaffold (from the cross-host reader)

- **CEB-22 — Worktree-safe durable state.** Store agent state *out of the repo tree* at `~/.compound-engineering`, keyed by `git rev-parse --git-common-dir` so all worktrees of a repo share it; three-layer cascade `local > project > global` first-found-wins; XDG fallback chain; "config is data, not a markdown document." Plus **migration-has-one-owner** (one skill migrates, one reports, entry skills warn — spreading it causes drift) and **deterministic-derivation-over-stored-config** (don't persist what a skill can compute). Candidate build-defaults / scaffold guidance.

### memory-discipline + personality product

- **CEB-23 — Auto-memory as evidence.** Read `MEMORY.md` before launching subagents, pass relevant entries as *supplementary evidence*; **a memory note contradicting a doc = a drift signal**; strictly read-only (writes go to team-shared structured docs, not machine-local memory); graceful absence (no dir → proceed). It draws *our exact* machine-local-auto-memory-vs-team-shared-repo-docs cut (`auto-memory-integration-requirements.md:10,29`). Relevant to our memory discipline, and to the personality product (auto-memory as a persistence substrate).

## New skill / workflow candidates (the discovery payload)

Genuinely new to us, surfaced per Rex's "inspiration for new skills/workflows" goal:

1. **`onboarding` skill** — crawl a repo → generate a human-facing `ONBOARDING.md` (5 first-hour-question sections: what is this / how organized / key concepts / primary flow / where to start); always full-regenerate (reading the old doc makes the agent do two jobs); existence is the only state. *Distinct from our CLAUDE.md/AGENTS.md scaffolding — it's a human-comprehension artifact we don't produce.* (`vonboarding-skill-requirements.md`)
2. **Iterative-optimization loop** (the shipped `ce-optimize`) — *feeds loop-graduation/chain-auto-compose as prior art*, but as a standalone skill concept (metric-driven self-improvement with immutable harness + judge) it's new to our catalog.
3. **Issue-grounded ideation / issue-intelligence agent** — theme-level clustering of tracker signal into strategic improvement ideas; independently dispatchable. (`issue-grounded-ideation-requirements.md`)
4. **Research-agent-over-a-data-source** (the Slack-analyst pattern) — a dispatchable read-only research sub-agent with a transferable mechanism: **MCP-availability-check (not tool-name check)**, two-level short-circuit (caller + agent-on-entry), bounded reads (cap ~3-5), parallel dispatch alongside other researchers, and a hard rule that the workflow **must not require** the integration. A template for surfacing any external-data context (Slack, Linear, Notion) into planning. (`slack-analyst-agent-requirements.md`)
