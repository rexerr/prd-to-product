# Cribs from the craft ecosystem (Pocock + Ciemborowicz)

Process and skill-authoring patterns mined from two MIT repos that work the same loop this workspace does — author a discipline, then compress it down to what actually changes an agent's decision. Same goal as the Steinberger sheet: **import scar tissue, not reinvent it**. These authors ran the skill-craft and rule-compression loops at higher volume and wrote down what broke; this sheet borrows their bugs without re-deriving them.

This is a **living adoption-tracker**, not a one-shot summary. Each crib has a stable `CF-NN` ID, the failure it prevents, the surface it lands on, and a status. Promote to a `D-NNN` decision (and an actual edit) when adopted. Read alongside `cribs-from-steinberger-ecosystem.md` (C-01–C-41) — where a craft crib extends an existing `C-NN`, it names it.

## The meta-theme

Both authors converged on one idea our cribs only touch at the edges:

> **A durable artifact is only worth its context cost if every line changes a decision.** Pocock names the test (the *no-op*: does this line beat the default?); Ciemborowicz operationalizes it as a compression method (keep only the decision-changing rule, trace what you dropped). The Steinberger sheet's "cite the failure mode" doctrine says *why* a rule earns its place — this lane supplies the vocabulary to *diagnose when it stops earning it*.

## Divergence / fit notes

Pocock's fleet is built for a **TypeScript engineer shipping product code into a GitHub issue tracker**, and most of its mass — `tdd`, `implement`, `to-issues`, `triage`, the Husky pre-commit setup — lives in exactly the world D-001 walls off: a running app with a test runner, a build step, and a multi-contributor inbound stream. That divergence is justified by product shape, not inertia: our "product" is markdown skills installed by symlink, one developer, direct-on-main. What ports cleanly is never the *code* skill but its *durable-artifact discipline* — the no-op test, the stale-path PRD rule, the red-capable-repro gate, the concept-keyed rejection ledger — because those are claims about what belongs in a markdown file that survives, and that is precisely what this workspace authors. Ciemborowicz diverges the other direction: his *method* (PROCESS.md, the conflict matrix) is a perfect fit for a rule-authoring loop, but his *output* (14 packs of SWE coding rules) is the out-of-domain content our generator is explicitly forbidden to emit. The recurring fit test across both authors: **adopt the discipline that governs a durable artifact; decline the tooling bound to a tracker, a build, or a codebase we don't have.**

## Round 2 — 6-lens primitive re-mine (Pocock skills, 2026-06-17)

The first pass triaged each of the 34 Pocock skills as a whole and through a single dominant lens, which is how it nearly lost CF-06 (de-dup-by-reference) — that primitive only surfaced because someone looked at the *handoff* mechanic rather than the skill's headline purpose. This pass decomposed every skill into its distinct mechanics and scored each independently across all six lenses (multi-surface / primitive-decomposition / shape-vs-content / meta-craft / composition-chaining / anti-pattern), on the premise that a 90%-off-domain skill can still donate one keeper.

The raw run minted 23 ids; a skeptical review consolidated them to **16 (CF-07–CF-29, gaps merged)** because the deeper rubric *fragments* — it split source-immutability across four ids and interview-discipline across three. The honest pilot finding is "real misses **plus** inflation," not pure yield, and ~6 of the keepers came from one squarely-on-domain skill (`writing-great-skills`). The standing consequence (see Delta): run the 6-lens pass **surgically** — only on skill-authoring/governance-domain skills — and keep single-lens whole-skill triage for the off-domain mass. New ids continue from CF-06; anything collapsing to an existing `C-NN`/`CF-NN` is parked under "Confirmed / already-held."
