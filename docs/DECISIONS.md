# Decisions log

Append-only log of significant technical and product decisions for prd-to-product.

> For currently binding constraints, see [`docs/DECISIONS_ACTIVE.md`](DECISIONS_ACTIVE.md). New decisions are appended here. Mirror to `DECISIONS_ACTIVE.md` only if they impose ongoing constraints not visible from reading code alone.

## What goes here

A decision belongs in this log if it is significant — meaning:

- it would surprise a future session that hadn't seen it made
- it ruled out an obvious alternative
- it affects a core workflow
- it touches a safety default

Do not log every small implementation detail. Log the calls that matter.

## Format

### [Short title]

**Date:** YYYY-MM-DD
**Context:** What situation prompted the decision.
**Decision:** What was chosen.
**Reason:** Why — what was ruled out and why.
**Revisit if:** Conditions that would make this worth reconsidering.

## Declining and parking

How a rejected or deferred concept is recorded so it is not silently re-litigated (CF-04, adopted as D-027):

- **Concept-keyed, not request-keyed.** One record per rejected *concept*, not per time it surfaced. A repeat proposal appends to the existing record's prior-occurrences; it does not mint a new one.
- **Durable WHY, never "too busy."** The reason must be a scope/philosophy call, a technical constraint, or a standing trade-off — not "no time now." A "too busy" rejection is a *deferral*, and **deferral ≠ rejection**: park it where it can re-enter (BACKLOG / a tracker's Parked section), don't record it as declined.
- **Read before re-proposing.** Before re-arguing a previously-cut concept, read its record; if a concept-similar record exists, engage its stated reason rather than re-opening from scratch.
- **Reconsideration supersedes the record.** If a declined/parked concept is later adopted, delete or supersede its record and cite the adopting `D-NNN` — never leave a stale "declined" beside a now-live thing.

This governs DECISIONS.md declines (e.g. D-008, D-013) and the **Parked** sections of the crib trackers, which apply the same shape to cribs. Stated here once and referenced there, not restated (CF-06, D-024).

### D-001 — Skills stay markdown-only (pending HTML investigation)

**Date:** 2026-05-10
**Context:** [`docs/html-over-markdown-brief.md`](html-over-markdown-brief.md) raises whether `prd-creator` and `design-system-bootstrap` should output HTML for richer artifacts. The investigation is parallel to the main skill-refinement work and not yet resolved.
**Decision:** All skills in this repo continue to produce markdown output until the HTML-over-Markdown investigation concludes and explicitly authorizes a change. The default holds; do not pre-decide based on speculation.
**Reason:** A switch to HTML touches output paths in multiple skills, downstream consumer assumptions, and the renderer story. The investigation is the right place to make that call. Switching ad-hoc would fragment the skills' output formats and orphan work in flight.
**Revisit if:** The investigation in [`docs/html-over-markdown-brief.md`](html-over-markdown-brief.md) recommends HTML for one or more skills, with a specific failure mode the change prevents.

**Date:** 2026-05-10
**Context:** Skill-refinement work is sequential and single-developer; each phase touches the same template and decisions files. Branching would force serial merges with no parallelism benefit.

**Reason:** Single developer, sequential work, files overlap → branches add friction with no win. Continuous-mode discipline (act on evidence, not speculation) applies.

### D-005 — Generators are non-destructive by default (prose-mitigated, not yet hook-enforced)

**Date:** 2026-06-08
**Context:** An LLM Council on improving the kit ([`docs/council/council-report-2026-06-08.html`](council/council-report-2026-06-08.html)) found that the kit's headline accuracy failure is the "qventus near-miss" — a generator clobbering a hand-authored design system — caught last time only by a human pause. Peer review sharpened it: this is a *write-guard* problem, not a diff/regression problem, and the file class at risk (`tokens.css` + seed components + `DESIGN_SYSTEM.md`) is written **only by `design-system-bootstrap`**, which wrote `tokens.css` unconditionally. `context-engineering` and `prd-creator` had no systematic existing-file guard either.
**Decision:** All three skills carry a **non-destructive write guard**: before writing any file, check if the target exists; if it does and is not a recognizable unfilled scaffold (no `<!-- PARAMETERIZE:` markers), show a diff and require explicit overwrite/skip consent (default skip). Merge is offered **only** where a merge operation is defined (DSB's `.claude/rules/design-system.md` snippet and `tailwind.config.tokens.ts`); whole-file artifacts (`tokens.css`, seed components, `DESIGN_SYSTEM.md`, `PRD.md`, `BRAND.md`, the context scaffold) are overwrite-or-skip. Skipped/overwritten files are reported in each skill's output summary with the standard markers `(skipped — already exists; not overwritten)` / `(overwritten with consent)`. Implementation: each skill's `generator/decisions.md` "Non-destructive write guard" section, its intake confirm-gate, output-summary, and SKILL.md gotcha (DSB also `principles.md`).
**Reason:** Closes the actual qventus path (DSB `tokens.css`) and brings ce/prd to parity for `PRD.md`/`CLAUDE.md` collisions (the field-society-demo collision). Cost-asymmetric: a missed re-generation costs minutes; one destructive run costs a day plus trust in the kit.
**Reliability ceiling (explicit):** this is a **prose** guard the agent must honor — it is **not yet enforced**. Per this repo's own thesis (prose rules don't enforce themselves; cf. `enforce_rules_as_hooks`), full closure requires a PreToolUse Write/Edit hook with session-aware consent state. **This decision mitigates the qventus class; it does not close it.** The hook is the load-bearing follow-up (see `BACKLOG.md`).
**Revisit if:** the enforcement hook ships (upgrade the framing from "mitigated" to "enforced"), or a real run shows the prose guard was dropped under context pressure (evidence the hook must be prioritized). **Update 2026-06-08: the hook shipped — see D-006; the qventus class is now enforced (interactive) / refused (headless), not merely mitigated.**

### D-006 — Write guard is hook-enforced (interactive-`ask` / headless-`deny`)

**Date:** 2026-06-08
**Context:** D-005 shipped the non-destructive write guard as prose and named its own ceiling: prose the agent can drop under pressure only *mitigates* the qventus clobber class. This is the load-bearing follow-up — the PreToolUse hook that enforces it. An earlier design (exit 2 + an agent-written consent token) was rejected: the agent under pressure could self-consent (forgeable). `permissionDecision: "ask"` forces the native dialog the agent cannot answer — the only non-forgeable primitive.
**Decision:** A global PreToolUse hook [`hooks/write-guard.sh`](../hooks/write-guard.sh) (matcher `Write|Edit|MultiEdit`), installed once in the operator's `~/.claude/settings.json`, enforces D-005 **while a generator run is armed**. Dormant unless armed (sentinel keyed by the payload's `session_id` — the safety key; a different project = a different session = dormant; refresh-on-match so a long idle run never ages out; TTL is GC only). Ladder: dormancy → control-path exempt → fresh-create (record as run-owned, allow) → run-owned (allow) → unfilled-scaffold marker (allow) → else **gate**. Gate = interactive `permissionDecision:"ask"` / headless·`bypassPermissions`·unknown `deny` (default-skip). "Pre-run" = existed before the run; **the hook maintains the run-owned set itself**, so a missing/empty owned-set fails SAFE (existing files gate, never silently allow). Each skill arms at run start / disarms at run end via Bash ([`generator/decisions.md`](../skills/design-system-bootstrap/generator/decisions.md)). See [`hooks/README.md`](../hooks/README.md).
**Verified (Claude Code 2.1.138 / 2.1.165 — probe + true end-to-end):** `deny` honored for Write+Edit in `default` and `acceptEdits` (bug #37210 not live); headless `ask` auto-resolves to a deny — does NOT hang (#9026/#64271 not live on the `claude -p` path); interactive `ask` surfaces a real dialog offering **allow-once only** (no for-session option → one approval cannot neuter the guard, confirmed by a per-file re-prompt); the interactive/headless signal is `CLAUDE_CODE_ENTRYPOINT` (`sdk-*` headless / `claude-desktop` interactive — `/dev/tty` and `permission_mode` were tested and **rejected** as discriminators). End-to-end: a pre-run file overwrite was blocked (in `permission_denials`, file unchanged) while fresh-create + run-owned re-edit passed; `.owned` maintained correctly. 15/15 unit assertions ([`hooks/write-guard.test.sh`](../hooks/write-guard.test.sh)). Caught: the hook must be executable (non-exec → exit 126 → tool is *allowed*); MultiEdit is not a tool in 2.1.138 (matcher keeps it for forward-compat).
**Scope of the claim (honest, no overclaim):** Enforced for the pre-run hand-authored class. Bypassable only by global permission-skip modes (`--dangerously-skip-permissions`); not by `[y]s` (no for-session option exists for hook-`ask`). **Headless functional ceiling:** an unattended run can **never** overwrite a pre-existing file — it `deny`s — so it is non-destructive *because* it is update-incapable on existing files (greenfield only). This clears the `/idea-to-product` orchestrator gate for the **clobber-safety class only**, not as "the orchestrator's full job runs unattended." Unprotected residual: a generator that forgets to arm (prose is the backstop); a forgotten owned-set fails safe.
**Not yet:** distribution-portable — the hook lives in the operator's `~/.claude/`, installed by hand; shipping it with the skills (a plugin install-time hook) is the future step.
**Revisit if:** a Claude Code version reintroduces the Edit-`deny` bug (#37210) or makes headless `ask` hang (#9026/#64271) — re-run [`hooks/write-guard.test.sh`](../hooks/write-guard.test.sh) + the probe; or the guard is shipped as a distributed plugin (upgrade the portability note); or `ask` is found to mutate session permission state (#39344/#37420) in a future version.

### D-007 — Skill-defined merges are agent-performed (diff-confirm), not handed to the user

**Date:** 2026-06-09
**Context:** Reviewing the D-005/D-006 write-guard, Rex found a related boundary that was backwards: design-system-bootstrap's `merge` paths told the agent to emit a snippet/merge-note and ask the *user* to apply it by hand — inverting "you approve, I apply." The behavior predated the write-guard work (initial commit `3fea33c`); D-005 only re-blessed it. It affected three DSB merge targets: `.claude/rules/design-system.md`, `tailwind.config.tokens.ts`, and `globals.css` (the third was an "emit a merge note" path the first fix draft missed).
**Decision:** Where a skill defines a merge operation, **the agent performs the merge itself and shows the diff for confirm — it never hands the user a snippet to apply.** Deterministic merge semantics avoid corrupting hand-authored files: the rule file gets a single HTML-comment-fenced block (`<!-- design-system-bootstrap:start/end -->`) appended at EOF on first merge, then updated in place — *positionally additive, not a semantic reconcile*; `tailwind.config.tokens.ts` appends only to `theme.extend.*`, never replacing existing keys; `globals.css` prepends the missing `@import`/`@tailwind` pieces at top-of-file (CSS drops a mid-file `@import`), each only if absent. **Interactive-only:** the merge `Edit` rides the D-006 hook (`ask` on the pre-run file) and the agent shows its own diff before writing. **Headless → skip:** the hook `deny`s edits to pre-run files unattended (D-006 functional ceiling), so the merge cannot run unattended and is reported as skipped. Implementation: DSB `generator/decisions.md` (`merge` branch + globals.css merge subsection + per-template table), `output-summary.md` (`(merged with consent)` marker, full merge set enumerated), `principles.md`, `SKILL.md`.
**Reason:** "You approve, I apply" — the agent does the editing. The old snippet path was safe-by-construction (no write) but pushed the work onto the user; this moves it onto the hooked `Edit` path. No-clobber is preserved **by the D-006 hook on the Edit** (probe confirmed `deny` honored for Edit, #37210 not live) plus the fenced/append-only determinism, **not** by refusing to write. Honest limit: the merge is positionally safe, not semantically reconciling — overlap with inline hand-written rules surfaces in the diff for the user to resolve.
**Revisit if:** a Claude Code version regresses Edit-`deny` enforcement (#37210) — the merge's safety on the Edit path would then rest on prose alone (re-run the probe + `hooks/write-guard.test.sh`); or a merge target needs true semantic reconciliation (the fenced-append model would no longer be sufficient).

Still **N=1** (same bundle) — the Rule-of-Two trigger above is unchanged. **Process note:** this council should have been preceded by reading D-008 ([D-027](DECISIONS.md) "read before re-proposing"); it wasn't, so the headline verdict duplicates the 2026-06-09 decision — net-new value was only the composition-fidelity dimension. ### D-009 — Recommend a council at genuine forks (recommend, never auto-run); reconcile bundle vs interview PRD by routing deltas by KIND

**Date:** 2026-06-09
**Context:** Two linked questions, run through an LLM Council ([report](council/council-report-2026-06-09-reconcile.html) / [transcript](council/council-transcript-2026-06-09-reconcile.md)). (Q1) How should the kit reconcile a Claude Design bundle's PRD — which often carries *superior* prototype-driven features/layouts — with a prd-creator interview PRD that is stronger on strategy/scope/rationale? (Q2) Should the LLM Council be wired in as a reusable *primitive* at the kit's decision points? The council was unanimous (Skeptic strongest 4/5, Long View biggest blind spot 5/5) and its peer round surfaced the load-bearing insights.
**Decision (Q2 — council placement):** The council is **NOT** a wired-in primitive. Instead, a *recommend-don't-auto-run* note lives in (1) this repo's [`CLAUDE.md`](../CLAUDE.md) session-discipline, (2) the scaffolded session-discipline of `context-engineering` (flat + modular templates + `examples/output-small`), and (3) a `prd-creator` principle for high-stakes scope forks. The note fires only when a decision is **both costly to get wrong and hard to reverse**; it recommends a council (or equivalent multi-perspective stress-test) before committing, and is phrased tool-agnostically in scaffolded output (the kit can't assume the plugin is installed).
**Decision (Q1 — reconciliation):** Build nothing structural (consistent with D-008). Reconcile by **human diff-and-adjudicate, section-keyed onto prd-creator's section skeleton, with deltas routed by KIND**: *pure additions* (a feature/layout the interview never spoke to) are presumptively safe to absorb; anything that *contradicts or re-expands a scope decision the interview made on purpose* is held as a **scope-gate** violation, framed to the operator as "the bundle re-introduces something you decided to cut" — not "the bundle has a better idea." This is the judgment-layer analogue of the existing write-guard hook (D-006).
**Reason:** The council killed the primitive on three independent grounds — ceremony on a lightweight single-dev kit, a hard dependency on an external marketplace plugin (the portability invariant the kit defends everywhere), and the reflexive tell that a "primitive" overreaching twice this week and saved only by peer review is a procedure to babysit, not a primitive. A *recommendation gated by a threshold* is the one council-in-the-system form the council endorsed (ad-hoc, hand-invoked at genuine forks). On Q1, the peer round showed options (a) and (d) were the same loop at different layers; the real risk is **scope-clobber** (prototype enthusiasm laundering deliberately-cut scope back in), plus a **self-adjudication bias** (the operator adjudicating is the same person who made the prototype) — both addressed by the KIND-routing + scope-gate framing rather than by automation.
**Honest limits:** scaffolding a recommendation into *every* generated project risks dead text where no council skill exists (mitigated by tool-agnostic phrasing + the tight high-stakes-AND-irreversible gate); the council also flagged that the strongest "this is superior" signal often lives in the bundle's *screenshots*, not its prose, so prose-merge tooling is the wrong place to invest. Encoding the KIND-tagging step into the kit is deferred until it's been done by hand ~3 times (Rule of Two/Three), not now.
**Revisit if:** the recommendation note proves to be dead text or ritual on real projects (loosen or gate it behind an intake question), or the KIND-tagging reconciliation is performed by hand enough times to justify encoding it as a scaffolded procedure.

### D-010 — prd-creator intake shifts from ask-anyway to draft-and-present when source material covers a cluster

**Date:** 2026-06-09
**Context:** prd-creator's cluster-0 rule (and the mirrored `principles.md:34` / `SKILL.md:32`) banned *silent* absorption of a pasted brief and, to do so, mandated asking every cluster's questions "anyway" ([`generator/intake.md`](../skills/prd-creator/generator/intake.md) line 25). A 2026-05-11 Squirreled validation run surfaced the overcorrection: a thorough brief covering modes/scope/V1 was re-interrogated cold ("give me a bullet list of capabilities"), and the user reasonably pushed back — "doesn't my brief tell you this?" `NOTES.md` had parked the "pre-fill-then-confirm" shape until ask-anyway "has run on a real project"; that gate is now met, and the negative result is the evidence that unparks it. Three sibling findings rode along (vocabulary leak, temporal hallucination, missing `## Gotchas`).
**Decision:** For each later cluster the cluster-0 source material **substantively covers**, the skill now **drafts that cluster's answer from the material and presents the draft for the user to confirm, edit, or correct** (draft-and-present); where the material is thin or silent, it asks cold (the original behavior). This is the sanctioned third behavior between **silent absorption** (banned, kept banned verbatim) and **ask-cold** (the overcorrection). Cluster 0 still runs first and still asks the source-material question. Bundled in the same pass: (1) internal scaffolding is scrubbed from user-facing copy — no "cluster N", no mid-capture "D-NNN candidate" narration, no "(from cluster N)" provenance tags — fixed at the spec source ([`output-summary.md`](../skills/prd-creator/generator/output-summary.md) Format block + rule 43, which contradicted its own "what never goes in the summary"); (2) a **preventive** no-temporal-claims guardrail (do not speculate about when source material was written); (3) the `## Gotchas` backfill (prd-creator was the only skill missing it after the 2026-06-01 audit). Implementation: `intake.md`, `output-summary.md`, `principles.md` (augmented line 34 + a new conduct section), `SKILL.md` (augmented + Gotchas), the three `examples/*/transcript.md`, and `NOTES.md` regression tests 11-12.
**Reason:** Honors the context-engineering Pass 1.7 lesson — which was against *silent* absorption specifically; draft-and-present is not silent, every draft is shown and signed off — while removing the friction of making a user re-state a thorough brief (which also reads as if the skill never read it). This is reversible prose backed by a real validation failure, so it sits **below the D-009 council threshold** (costly-but-reversible); a council was considered and declined. Distinct from D-009's prd-creator hook (the council-recommendation principle for high-stakes scope forks) — different principle, no collision.
**Verification:** Regression tests 11 (draft-and-present on covered clusters / ask-cold when thin — exercised by medium+large with a brief, small without) and 12 (broadened-pattern assertion: zero `[Cc]luster [0-9]` in user-facing copy, `D-0[0-9]` only inside the cluster-5 read-back and output blocks). Test 12 asserts on the category pattern, not literal strings, deliberately — a literal-string check would re-encode the blind spot that let "Captured as D-002" slip past an earlier hand-enumeration. `examples/small/PRD.md` is unchanged (the edits are interview-conduct and phrasing, not output shape).
**Propagated 2026-06-09 (same day):** the vocab + temporal halves landed in the two sibling skills — `context-engineering` (fixed one user-facing quoted prompt that named "cluster 0 source") and `design-system-bootstrap` (no spoken leak existed) — plus a preventive no-temporal-claims guardrail in each cluster 0 and a shared "Interview conduct" principle pair in each `principles.md`. The investigation confirmed the behavioral half was a **no-op**: both siblings already prescribe draft-and-present ("present extracted answers, confirm or correct, rather than ask cold"), and their example transcripts use `## Cluster N` as structural headers (model-facing documentation), not spoken leaks — left untouched, consistent with leaving prd-creator's NOTES/README. 20 lines, 4 files. ### D-034 — CF-23 adopted: the two-load model governs `disable-model-invocation`

**Date:** 2026-06-18
**Context:** Wave-2 Sprint 1, Pass C. `CF-23` ([`cribs-from-pocock-craft.md`](cribs-from-pocock-craft.md)) names the tradeoff behind every skill's invocation-mode choice — previously made ad hoc with no stated rule. This record *is* the binding rule (not a separate "adoption" entry); [`skill-md-template.md`](skill-md-template.md) carries only a pointer/example.
**Decision (the rule):** every skill split spends exactly one of two conserved costs. **Context-load** — a *model-invoked* skill's description sits in the context window every turn, costing tokens and diluting attention whether or not it fires. **Cognitive-load** — a *user-invoked* skill (`disable-model-invocation: true`) costs nothing in-window but is indexed only by the human's memory, so it's findable only if the human remembers it exists. **Set `disable-model-invocation: true` when** the skill is invoked deliberately at known moments (explicit-invoke by nature) and auto-selection isn't the point — its always-in-window description would tax every turn for a skill rarely the right pick (e.g. `furnace-plan`, `frontend-er`). **Leave it model-invoked when** firing on trigger phrases *without* the user remembering the skill is the whole value (e.g. the `context-engineering`/`prd-creator`/`design-system-bootstrap` chain). When unsure, default to user-invoked for narrow/heavyweight tools (protect the window) and model-invoked for broadly-useful ones.
**Reason:** A forward-looking authoring constraint with a live consumer — the queued `/repo-miner` shape decision turns on exactly this cut. Names the cost C-12 only gestured at. **Mirrored to `DECISIONS_ACTIVE.md`** — it binds future skill-authoring and is not visible by reading any single skill (the one Sprint-1 crib that meets the ACTIVE bar); marker bumped D-033 → D-034.
