## Framed question

## Advisor responses (de-anonymized)

### The Skeptic

Option D — no new skill, handle foundation ad hoc — the only choice that survives the repo's own rules. Building any foundation skill now violates Rule-of-Two; the GAP is theoretical (one shipped Swift app = sample size of one, zero repeated foundation failures). The fatal flaw in A/B/C is staleness, worse than skill-elegance: a foundation skill freezes one snapshot of engineering judgment (Expo SDK version, auth library, CI config, dependency pins) into markdown a non-engineer cannot audit; when it rots in ~6 months Rex can't see it's wrong and will trust an outdated security baseline because he's delegating the judgment to verify it. Bare Claude, invoked per project, queries current best practice at build time — the skill actively makes the outcome worse for the one user who can't catch the error. C is the trap dressed as rigor: expo-foundation AND swift-foundation AND whatever's next, each rotting independently under a lone non-engineer. B is worst — it knifes the D-001 invariant that gives the whole product coherence, for a gap you haven't proven hurts. The real test: build foundation the ad-hoc way three times, log what Claude redoes identically each time. If a stable, non-stale pattern emerges, THEN scaffold that checklist — and only that. Don't scaffold the judgment; scaffold the checklist, if one even survives contact with reality.

### The Architect

Foundation work isn't one job — it's two glued together by a false assumption. Job one: pick the right stack and decree the architecture (judgment, irreversible, high-stakes). Job two: run init, wire auth, set up CI (mechanical, deterministic, low-judgment). You're treating "fill the gap" as a single skill because the gap looks like one hole. It isn't. Once split, the answer falls out. The judgment half is already a markdown artifact — a stack-decision record, a security baseline checklist, an architecture-of-record. That's exactly what context-engineering shaped (CLAUDE.md, rules, docs) and it does NOT break the markdown-only invariant. The execution half — actual npx create-expo, dependency installs — is what Claude already does well ad hoc, per project, with current knowledge no frozen skill can match. A skill that hardcodes "the 2026 secure Expo setup" rots the day Expo ships a new default. So: not A, not B, not C. The real answer is D-plus-a-markdown-artifact. Extend context-engineering (or a thin sibling) to emit a FOUNDATION.md — chosen stack + why, security baseline, QA gates, CI expectations, init commands as a checklist Claude executes live. Markdown decides; Claude's live session does. No invariant broken, no stack-specific proliferation, no frozen engineering judgment. Smallest experiment that proves it: next real project, have Claude hand-write that FOUNDATION.md and execute from it once. If it reliably produces a sound foundation, THEN extract the skill — Rule-of-Two satisfied by evidence, not hunch. You don't have a skill-shaped problem yet. You have one un-run experiment.

### The Stranger

Stack-agnostic stack selection is the load-bearing phrase that hides the problem. "Claude picks the right stack per job" sounds like delegated expertise, but a skill is frozen text — the moment you write down "for mobile, prefer Expo/React Native; wire auth like this," you've hardcoded a 2026 snapshot of a field that moves quarterly. The expertise Rex thinks he's encoding is exactly the thing that rots fastest, and he — a non-engineer — is the least equipped person to notice when the security baseline or auth pattern in his own skill has gone stale and started shipping vulnerabilities into his apps. A code-writing skill maintained by someone who can't read the code is a liability that compounds silently. Who's actually affected: Rex, six months from now, generating a "foundation" he trusts because his own skill produced it — and the users of whatever app he ships on top of a CI/auth/security setup nobody competent re-validated. Words doing unexamined work: "correct, secure, tested foundation" assumes there's one right answer per stack. There isn't; that's why it's judgment, not a template. The tell against building anything: the repo's own Rule-of-Two says build on repeated real need. Rex has shipped one app. He has a [GAP] in a diagram, not a documented pattern of pain. That is a hunch wearing a flowchart. Pick D. Build foundations ad hoc with live Claude, which re-derives current judgment every time instead of freezing it. Let the repeated friction — the same setup steps, hand-done three times — write the spec for you. If a pattern emerges, capture it then, stack-specific (C), as a checklist Claude executes, not as Claude's brain in a jar.

### The Long View

Stack-specific skills (option C) is a trap that's invisible today and obvious in three years: you'd be signing a lone non-engineer up to track the security baseline, auth patterns, and CI conventions of Expo and Swift and whatever's next — as each one drifts underneath you. Encoded engineering judgment is the fastest-depreciating asset you can put in this repo. A "correct, secure" foundation written in 2026 is a liability by 2028; Claude's live judgment isn't. You'd be freezing the one thing that should stay liquid. That's the second-order effect hiding behind the gap: the gap isn't a missing skill, it's a desire to pin down expertise you've deliberately chosen to delegate. Every line of stack-specific foundation code you commit is a maintenance debt that compounds while the underlying model that could've just done it fresh keeps getting better. You'd be racing the platform — and losing. What compounds in your favor instead: the markdown-only invariant (D-001) and the anti-speculation discipline. Those are reputation-grade assets — they're why this toolkit stays legible to you, a non-engineer, three years on. Option A or B spends that identity to buy a depreciating artifact. Bad trade. The move: D now, and let the gap teach you. Handle foundation ad hoc with Claude, but instrument it — keep a thin markdown log of what Claude picks and where it stumbles. If after several real builds the same failure recurs, you'll have earned (per Rule-of-Two) a markdown skill that encodes foundation judgment — a checklist, a decision tree, security questions to ask — not foundation code. That honors the invariant and captures the only thing worth capturing: the repeatable judgment, never the perishable output.

### The Veteran

The obvious move here is D — handle it ad hoc — and the standard playbook usually says exactly that. I've watched a dozen toolkits die of premature skill-proliferation: someone builds an "any-stack bootstrap" skill, it encodes their 2024 opinion of Expo, and eighteen months later it's confidently wiring up deprecated auth patterns nobody caught because a markdown skill can't fail a test. That's the pattern, and it's brutal for a lone maintainer. A code-writing foundation skill is a liability that ages, unlike your markdown skills which age into mild staleness, not broken builds. So why isn't D just the answer? Because the Rule-of-Two cuts the other way: this isn't a hunch. The [GAP] is real, repeated, and named — every project hits it. That's earned, not speculative. But "the need is real" does not license "build a code skill." The need is judgment at the foundation step, and judgment is the one thing that rots fastest when frozen into a generated scaffold. The case that's special: your skills are markdown because markdown survives. The moment a skill emits product code, it inherits a maintenance contract — test it, secure it, version it against the stack — that a non-engineer solo maintainer cannot honor. That's not skill design, that's a different product (B is the category error; it poisons the one skill that works). The honest move: A in spirit, but the skill should emit a stack-decision-and-foundation plan — markdown, checklists, a security/QA baseline Claude executes live — not code. Keep markdown-only. Let Claude write the code each time with current knowledge. Don't freeze the judgment; scaffold the questions. Reject C — stack-specific skills are the proliferation trap dressed as rigor.

## Chairman synthesis

**Don't build a foundation skill yet — capture the judgment as a hand-written FOUNDATION.md on your next real project, let live Claude execute it, and extract a skill only if a stable pattern recurs.**

### Where the council agrees

The convergence is unusually strong, and that matters. All five advisors independently rejected A, B, and C — nobody defended a code-writing skill in any form. Shared reasoning: foundation work is the fastest-depreciating thing you could freeze in this repo. Stack conventions, auth libraries, CI config, and security baselines move quarterly; a 2026 "secure" foundation is a 2028 liability, and you — a non-engineer — are least able to spot when your own skill's security baseline has silently gone stale. Live Claude queries current best practice at build time; frozen markdown (or worse, code) cannot. Four of five also converged on the constructive half: the judgment part (chosen stack, why, security baseline, QA gates) is legitimately a markdown artifact that does NOT break the markdown-only invariant — and all five reviewers independently ranked the Architect's judgment-vs-execution split as the strongest single contribution. "Hand-write it once, extract only if it recurs" is the agreed path through Rule-of-Two.

### Where the council clashes

The only real fault line is timing, not direction: the Skeptic/Stranger's PURE D (handle it ad hoc, write nothing structured until friction forces it) versus the Architect/Long-View/Veteran's INSTRUMENTED D (produce a markdown FOUNDATION.md now as a deliberate decision-record-and-checklist). The Skeptic would wait for three real builds before writing anything reusable; the Architect would write the artifact on build one precisely so the Rule-of-Two evidence can accrue. Everything else — reject A, reject B as a category error, reject C as multiplying the rot — is unanimous.

### Blind spots the council caught

The peer round surfaced two gaps no single advisor caught. First, load-bearing: every advisor asserted "Rex shipped one app, so the gap is a sample-size-of-one hunch" — and nobody verified it. The entire anti-build case (Rule-of-Two says don't build on one data point) rests on that headcount, lifted straight from the prompt. But the counter-reading is just as available: the [GAP] is a named, designed node every project structurally hits — closer to "earned repeated need" than "one-off hunch." The frequency of the gap is the hinge of the whole decision, and it is currently unestablished. Second, the unauditability problem may just be relocating, not dissolving: the Architect's move assumes a markdown FOUNDATION.md + live Claude reliably produces sound auth/CI/security, but a non-engineer can't audit THAT output any better than frozen code — so "markdown keeps judgment liquid" might move the can't-verify-it problem one step downstream. (One reviewer's counter: a plain-English checklist's staleness is at least auditable in a way code's isn't — real, but it doesn't establish the executed foundation is sound.) The recommendation holds regardless of the headcount, but the headcount sets how fast you move from "ad hoc" to "skill," so step one must resolve it.

### The recommendation

Go with INSTRUMENTED D — the Architect's split, not the Skeptic's pure ad-hoc. Reject A and C outright (a code skill or per-stack skills are depreciating maintenance debt a solo non-engineer cannot honor; B is the same error twice). The reason to side with the instrumented majority over pure-D is that pure-D has a quiet failure mode the reviewers exposed: with no artifact, the Rule-of-Two evidence never accrues — you'll hit the gap three more times, solve it three different ways from scratch, and still have nothing to extract. Writing a hand-made FOUNDATION.md on your next real project costs almost nothing, breaks no invariant (it's a decision record — the shape context-engineering already emits), and IS itself the instrument that tells you whether a skill is ever warranted. Critically, it captures judgment as a checklist Claude executes live with current knowledge: you are not freezing the auth library, you are freezing the QUESTION "is auth wired to current best practice?" That question doesn't rot; the answer is recomputed every session. That's the whole trick, and it's why it respects both the markdown-only invariant and your delegated-engineering-judgment posture.

### The one thing to do first

Before writing any skill or even the FOUNDATION.md template, verify the load-bearing premise: count how many times you've actually stood up a technical foundation and whether the same things genuinely went wrong each time. One Swift app is one data point; if React Native makes two and you can name a repeated failure, the gap is earned and the FOUNDATION.md experiment starts on your next build. If it's truly a single past instance, you write nothing reusable yet — you hand-author the FOUNDATION.md on the next real project as the first deliberate data point, and only recurrence decides the skill.
