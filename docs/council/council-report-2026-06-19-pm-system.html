Council Report

Ticket + queue vs. the dense single-file BACKLOG

5 advisors · anonymous peer review · chairman synthesis

The question

Should Rex — a designer running a SOLO skill-development repo whose main collaborator is an AI coding agent (Claude Code), no team, no deployed app — replace his current project management (a single dense BACKLOG.md where each item is an 800+ word entry fusing a prioritized 'what's next' list WITH a deep reasoning archive, read whole by the agent at every session start) with an industry-standard ticket + queue model (per-file tickets, a structured file, an HTML app, or Linear/GitHub Issues)? Or is the dense single file actually well-suited to a solo-human-plus-agent workflow, making tickets pointless ceremony? Constraints: a standing invariant (D-001) requires markdown for context files the agent reads at session start; CLAUDE.md warns that relocating docs breaks ~75–90 cross-references.

The recommendation

Council verdict

Keep one markdown file but split it in two layers — a thin top-of-file priority queue over the dense reasoning archive — and do not adopt tickets, an external tool, or the HTML app.

Restructure BACKLOG.md in place: a thin, priority-ordered queue at the top (one line per active item — title, status, next action, and a link to the detail), with the dense reasoning kept below it in the same file. Do not adopt tickets, do not move to Linear or GitHub Issues, and do not build the HTML roadmap app. The reasoning is straightforward: the only constraint that actually governs this repo is the agent's session-start read, and the genuine defect is paying archive-weight tokens to extract a queue-weight signal every single session — a tax that compounds as the archive grows. The two-layer split fixes exactly that without triggering any of the repo's known hazards: keeping the archive in the same file means anchors don't move, so the ~75–90 cross-references keep resolving; staying markdown honors D-001; and adding no UI surface avoids building product to manage a workspace. This is a 30-minute, fully reversible edit — which also defuses the Long View's false urgency: you're not racing a closing window, you're making a cheap change whenever you like. Two disciplines make the split durable rather than cosmetic: (a) the queue links into the existing DECISIONS.md/retros records rather than restating them, so you get a pointer not a third copy; and (b) you add a one-line rule (in CLAUDE.md or atop BACKLOG) that queue entries stay one line and reasoning lives below or in the archive — the guardrail against silent re-fusion.

The one thing to do first

Before editing anything, measure the actual session-start token cost of reading BACKLOG.md today and confirm whether its reasoning is already duplicated in DECISIONS.md/retros — that one check tells you whether this is a restructuring job or a deduplication job, and gives you the before/after baseline the repo's own verification discipline requires.

Where the council agrees

The council is close to unanimous on the destination. All five reject the ticket-and-queue model outright: it solves coordination problems (handoffs, assignment, multi-human status visibility) that a solo-human-plus-agent workflow simply does not have, so it imports ceremony with nothing to coordinate. All five reject the external tools (Linear, GitHub Issues) for the same reason plus a sharper one — they live outside the repo, so the agent can't read them at session start without a tool call, which collides with D-001. All five reject the HTML app as ceremony in a costume that also fights the markdown-only invariant. And four of the five converge on the real diagnosis: the defect was never the file count, it's the fusion of two artifacts with opposite read-frequencies and lifespans — a perishable, every-session queue welded to a rarely-read, append-only reasoning archive — so the agent pays archive cost at queue frequency. The prescription that follows: thin queue on top, dense archive below or linked.

Where the council clashes

The one real fracture is the Veteran versus everyone else. Four advisors call the fusion the bug — the thing to fix. The Veteran calls the fusion the feature, arguing that an agent resuming cold works better with reasoning adjacent to action, so reasoning should stay inline. Four of the five peer reviews independently flagged this as the Veteran's blind spot: a thin index added above still-inline dense entries doesn't stop the archive from being re-read every session — it adds signal without removing the tax. The Veteran is right that the fix is cheap and reversible and that 'professionalizing' is the trap; he's wrong that inlining is costless. The other latent clash is on urgency: the Long View insists migration is 'cheapest today and never cheaper,' which a reviewer rightly notes overstates the case for what everyone agrees is a low-risk, reversible edit.

Blind spots the council caught

The reviews surfaced three things no advisor handled, and they matter more than the verdict itself:

1. Nobody measured the premise. Every advisor asserts the archive is 're-read at full token cost every session,' but none verified whether the agent actually ingests all 800-word entries or could simply be instructed to read only the queue header. The repo's own 'verification before claiming done' discipline demands measuring session-start tokens before and after — yet no advisor proposed it as the gate.

2. The split could spawn a third copy of the 'why.' The reasoning may already be duplicated in DECISIONS.md and retros (which the BACKLOG cross-references). If so, the fix is partly deduplication, not restructuring — and a careless split risks creating a third home for the same reasoning instead of a pointer into the existing one.

3. The cross-reference hazard is real and was hand-waved. CLAUDE.md warns that relocating docs breaks ~75–90 cross-references. Advisors who claimed 'no cross-reference breakage' asserted it rather than checked it. The safe version keeps the archive in the same file (queue on top, archive below) so anchors don't move — moving the archive out is where the breakage lives.

4. No advisor named the anti-re-fusion discipline. What stops the agent from re-bloating one-line queue entries back into 800-word prose over time? Without a rule, the split silently undoes itself.

Advisor responses

01 The Skeptic

Tickets-per-file fails the one constraint that actually governs this repo: the agent's session-start read. The whole BACKLOG advantage isn't "one file" aesthetically — it's that a single read gives the agent the priority ordering plus the reasoning in one shot. Split into N ticket files and the agent now needs an index to know what's next, then must fan out reads to reconstruct the why. That's strictly more tokens and more round-trips, and it reintroduces the exact failure CLAUDE.md keeps legislating against: "lost context between sessions." You'd be optimizing for a coordination problem you don't have — there is no second human to assign a ticket to.

Now turn the skepticism on the status quo, because it's not clean either. The real defect isn't the file count; it's the fusion. An 800-word entry that welds "do this next" to a decision archive means the actionable signal is buried in prose the agent re-pays for every single session, forever, whether or not today's task touches it. That's a slow tax, and it compounds as the archive grows.

But the HTML-app fix is a trap. It collides head-on with D-001 (markdown-only for session-start context) and the markdown-only architecture invariant — the agent can't cheaply read a JSON island, and you'd be building product to manage a workspace. Drag UI for a one-person backlog is pure ceremony.

The honest answer: don't migrate, split the two jobs inside markdown. A thin top-of-file queue (one line per item, priority-ordered, pointer to detail) plus the archive below or in docs/. Cheap, reversible, no cross-reference breakage, no D-001 fight. Tickets solve a team problem you don't have.

02 The Architect

Tickets versus a dense BACKLOG is the wrong axis. You're treating this as a project-management format choice when it's actually a context-engineering problem: the failure mode isn't "no queue," it's that you've fused two artifacts with opposite read-frequencies and opposite lifespans into one file. The actionable queue is read every session and changes constantly. The reasoning archive is read rarely, on demand, and is append-only. Cramming them together means the agent re-ingests an 800-word/item archive every session to extract a one-line "what's next" — you're paying archive cost at queue frequency.

So don't import the ticket model. Import the one property that makes tickets work and that you're missing: separation of the live queue from the dead record. The simplest first move that proves this: split BACKLOG.md into a thin top section (one line per active item: title, status, next action, link) and push the prose reasoning down or out into the retros/decision records you already cross-reference. The agent reads the thin queue every session; it follows a link into the archive only when it actually picks that item up. No new tool, no per-file ceremony, no migration of cross-references — they keep resolving.

Skip Linear/GitHub Issues entirely: they live outside the repo, so the agent can't read them at session start without a tool call, and they break D-001's "markdown the agent reads at startup." Skip the HTML app too — it's solving a human-drag-UI problem you don't have as a solo operator, and it violates the same invariant.

The variable to optimize is session-start tokens per decision, not "do I have tickets." Split the file; measure the read.

03 The Stranger

Per-ticket files would force the agent to do something it is bad at: discover which files matter before reading them. Right now the agent reads one file and has the whole board plus the reasoning behind it. Split that into thirty ticket files and the agent either reads all thirty anyway (same tokens, more file-open overhead) or guesses which three are relevant and resumes work on a wrong or stale assumption. The single-file read is not legacy mess — it is the format that matches how this particular collaborator ingests context.

But there is a real fault line buried in the framing. "Fuses two jobs" is the tell. A prioritized what-next queue is read every session; an 800-word nested reasoning archive is read once and referenced rarely. Co-locating them means the agent pays the archive's token cost on every single resume, forever, to get the queue. That is the actual problem — not "no tickets."

What outsiders are assumed to already accept: that "industry standard" carries authority here. It does not. Linear and GitHub Issues exist to solve a problem this workspace does not have — multiple humans needing shared visibility, assignment, and status they did not personally write. Solo-plus-agent has none of that. Importing the tool imports the ceremony and the coordination tax with no coordinating to do.

Who is actually affected: two readers, Rex and the agent. Neither needs handoff. Both need fast resume.

Verdict: do not adopt tickets. Split the queue from the archive — a lean actionable queue at the top, the reasoning history in a separate referenced file. That is the cheap, reversible win. The HTML app is ceremony wearing a different costume; it also breaks D-001.

04 The Long View

Reading this whole file at session start is the load-bearing fact, and it's the one that compounds against you. BACKLOG.md grows monotonically — every retro, every decision, every cross-reference accretes. In three years a fused reasoning-archive-plus-queue isn't an 800-word-per-item file, it's a document where the actionable "what's next" is a thin signal buried in a deep sediment of why-we-did-things. The agent re-reads the sediment every session, at rising token cost, to extract a queue that could have been 200 lines. That's not a steady state; it's a slow leak that gets worse precisely as the project succeeds.

The thing invisible from where you stand: the two jobs depreciate at opposite rates. The queue is perishable — today's priority is worthless next month. The reasoning archive appreciates — old decisions get more valuable as context for new ones. Fusing them means your perishable layer drags your durable layer through a full re-read every session, and your durable layer bloats the file you scan to decide what to do next. They want to live apart.

What this makes true later if you split: a thin queue (the perishable "now/next") plus the archive you already have (DECISIONS.md, retros — your durable layer, untouched). The queue stays cheap to read forever; the archive stays linked, not inlined.

Don't build the HTML app. A JSON island violates the spirit of D-001, adds a UI surface a solo operator maintains forever, and the drag-to-reprioritize ceremony is exactly the human-coordination theater an agent doesn't need. The depreciating asset (software you maintain) to manage a queue is the wrong trade.

Reversibility cuts toward acting now: cross-references get harder to untangle every session you wait. Migration is cheapest today and never cheaper again.

05 The Veteran

The dense single-file BACKLOG is the right tool, and the instinct to "professionalize" it is the trap. I have watched this exact migration a dozen times: someone imports the ticket-and-queue ritual because it's what "real teams" do, and six months later they have a Linear board nobody grooms, tickets that rot because the context lives somewhere else, and a workspace that now requires two reads to do one thing.

Ask why the obvious move is obvious. Tickets exist to solve coordination problems you do not have: handoffs between people, status visibility for a manager, parallel work that must not collide, the need to assign. You are one person and one agent on `main`. Every cost the ticket model amortizes across a team, you pay alone with zero payoff. That's not "industry standard transferring" — that's cargo cult.

The genuine difference here: your collaborator reads the entire file every session and works better with reasoning adjacent to action. Splitting reasoning-archive from actionable-queue — the thing tickets do — is precisely wrong for an agent that resumes cold and needs the "why" inline. Per-ticket reads multiply context cost and lose cross-reference locality. The fusion you're uncomfortable with is the feature.

Where the standard playbook does apply: the actual pain isn't structure, it's that prioritization is buried in 800-word prose. Fix the queue, not the model. Put a thin prioritized index at the top of BACKLOG — five lines, "what's next" — and leave the dense entries below as the archive. That's a 30-minute edit, fully reversible, breaks zero cross-references. Kill the HTML app outright: it fights D-001 and buys you drag-UI you don't need.
