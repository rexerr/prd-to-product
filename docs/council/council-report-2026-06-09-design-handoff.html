Council Report

Adopting Claude Design handoffs: new skill, mode, or neither?

5 advisors · anonymous peer review · chairman synthesis

The question

Should the prd-to-product skill kit add a NEW SIBLING SKILL (design-handoff-adopt) to ingest Claude Design handoff bundles (HTML/CSS/JS prototypes + an existing token-CSS design system + screenshots + sometimes a PRD) and codify them into the kit's conventions? Alternatives: (B) a second mode inside the existing design-system-bootstrap skill; (C) no new capability, handle each bundle as a one-off; (D) defer until a second bundle appears. The kit is a single-developer workspace of single-purpose, markdown-only skills that don't auto-compose (chained by hand; an /idea-to-product orchestrator is proposed but unbuilt). The mechanical part is token transcription (parse :root CSS custom properties into a two-tier tokens.css, preserve oklch() exactly, Tailwind off); the judgment parts are multi-aesthetic reconciliation and pixel-perfect component recreation. Claude Design ships its own README instructing implementing agents. Pressure-test: is 'extract/transcribe, don't generate' a repeatable skill procedure or bespoke judgment? Does a 4th node deepen the chain-composition weakness? What's the honest ceiling?

The recommendation

Council verdict

Don't build a fourth skill — and before building anything, run a one-hour timed hand-transcription of the real bundle to prove the 'mechanical' token-adopt is actually lossless.

Do not build design-handoff-adopt as a fourth sibling skill — that is unanimous and correct. Side with the Architect's decomposition gated by the Skeptic's fidelity test, the strongest combined reasoning. Sequence: (1) Resolve the ingest-at-all question first, empirically. Before writing any artifact, hand-transcribe the real bundle's :root into the kit's two-tier tokens.css and answer one question: is it lossless? If the bundle's oklch/motion/font structure does not fit the two-tier shape without mutation, stop — do not adopt; leave the bundle authoritative, let the kit reference it, and govern only what the kit already owns (write guard, doc structure, rules). A lossy 'adopt' is strictly worse than using the bundle as-is. (2) If and only if transcription is lossless and fast, build the mechanical 20% as a slash command under the write guard — not a skill, not a design-system-bootstrap mode; a command is the smallest artifact and sidesteps both the Veteran's command/mode conflation and the routing-mode debt. Scope it to: parse :root, preserve oklch() verbatim, fold into two-tier tokens.css, flip Tailwind off. Leave reconciliation as an advisory prompt and recreation delegated to the bundle's README in guided prose. (3) Regardless, do not design a reconciliation 'procedure' from this one bundle (N=1 hazard); log what was genuinely mechanical and treat the re-adoption / diff-merge lifecycle as an open question for bundle #2. Net: closer to defer-then-narrow-to-a-command than to building anything now — defer the procedure, but run the cheap test that converts the defer into a confident build-or-don't.

The one thing to do first

Open the real Claude Design bundle and hand-transcribe its :root custom properties into the kit's two-tier tokens.css right now, on a timer. You are testing one thing: does every token (including oklch, motion, font) land losslessly in the two-tier shape? Lossless and under ~30 minutes → build the token-adopt slash command. Lossy or fighting the structure → do not adopt; keep the bundle authoritative and have the kit reference it.

Where the council clashes

Largely unanimous on direction; two genuine fault lines remain. First: build-the-mechanical-core-now (Architect, Veteran, Long View — a narrowed mode or command) vs. defer-until-bundle-2 (Skeptic, Stranger — D/C). The build camp says the token-parse step earns its keep today because it can't drift or hallucinate a palette; the defer camp says designing any reconciliation procedure from exactly one bundle (N=1) hard-codes that bundle's quirks as 'the format.' Second, and deeper: should the kit ingest the tokens at all? Blind spots the council caught

The peer reviews surfaced gaps every advisor missed or under-weighted. (1) The fidelity / lossy-isomorphism risk (caught by four of five reviewers): even the 'safe' 20% may be lossy. If the two-tier tokens.css cannot losslessly hold the bundle's oklch/motion/font structure, transcription itself is the mistake — it manufactures drift between two sources of truth, and the 'build the mechanical core' verdict rests on that unproven premise. (2) The versioned-bundle re-adoption lifecycle: every advisor treats the bundle as a static artifact transcribed once, but the real recurring cost is re-adopting an updated bundle into an already-edited tokens.css — a diff/merge problem, not a one-time extract — and whether 'preserve oklch, Tailwind off' is even stable across Claude Design releases decides whether the mechanical core is a durable script or itself bespoke. (3) The unexamined premise that the kit should own the tokens at all: Claude Design ships a working prototype plus its own README, so the highest-value move may be to consume the bundle natively and let the kit govern only what it already owns. (4) The Long View's smuggled premise: it justifies mode-over-sibling by appealing to an orchestrator the kit has explicitly not built — possibly a phantom requirement.

Advisor responses

01 The Skeptic

"Extract, don't generate" is not a skill — it's a transcription chore wearing a skill costume. What actually varies run to run: which aesthetic is a theme variant vs. dead-end; whether a screenshot is canonical or scratch; how a bespoke component decomposes into token-wired primitives; what the PRD says to keep. Every one is human judgment Claude Design's own README already steers. The repeatable 20% (copy :root tokens verbatim into two-tier tokens.css, flip Tailwind off) is a 30-line slash command, not a fourth skill.

Three failure modes nobody's naming:

1. Fidelity trap. "Preserve values exactly" sounds safe, but the kit's two-tier token shape will not be isomorphic to a 40-token oklch/motion/font system. The moment you map their structure onto yours, you've mutated it — and a lossy "adopt" is strictly worse than just using the bundle as-is, because now there are two sources of truth drifting.
2. N=1. You're designing a reconciliation procedure for "multiple aesthetics" from exactly one bundle. Build the abstraction now and you'll hard-code this bundle's quirks as "the format."
3. Chain debt. A fourth non-composing node in a chain whose headline weakness is non-composition.

Honest ceiling: reliably copies tokens and sets a flag. Everything distinctive stays human. Verdict: D, leaning C. Defer. Do the next bundle by hand against the README, log what was genuinely mechanical, and only then decide if there's a skill-shaped residue.

02 The Architect

"Adopt vs. bootstrap" is the wrong axis to optimize. Strip it down: what you actually have is a tool-handoff problem. Claude Design already emits a README telling agents how to implement the bundle. The kit's job is not to re-implement that README as a skill — it's to translate the bundle into the kit's conventions so the rest of the kit (rules, docs, write guard) can grip it. That's a much smaller, more repeatable thing than "recreate components pixel-perfect."

So decompose the proposed skill. It bundles three jobs of wildly different repeatability:

1. Token transcription — deterministic. Parse :root, preserve values, fold into two-tier tokens.css. This is a real, repeatable procedure. A skill earns its keep here.
2. Multi-aesthetic reconciliation — pure human judgment. No skill makes "are these theme variants or competing directions?" repeatable. The skill can only surface the fork and ask.
3. Pixel-perfect component recreation — that's Claude Design's README's job, not yours. Duplicating it is the trap in the question.

So the honest answer is B, narrowed — an "adopt" mode in design-system-bootstrap, scoped to job #1 only, with job #2 as a prompt and job #3 delegated to the bundle's own README. Bootstrap already owns tokens.css shape; an inverse path (preserve vs. generate, Tailwind off) is one branch, not a fourth chain node.

Smallest experiment that proves the variable: take the real bundle, hand-transcribe its tokens into the kit's tokens.css right now, by hand, timing yourself. If it's mechanical, build the mode. If you keep hitting judgment calls, the skill was never the leverage — the README was. Don't build until that 30-minute test runs.

03 The Stranger

"Claude Design," "handoff bundle," "two-tier tokens.css," "the qventus class," "non-destructive write guard," "oklch accents" — these are private vocabulary. The decision is framed as if a reader already agrees that the kit's conventions are worth conforming to, that "single-purpose skills" is a virtue rather than a tax, and that a bundle from Claude Design should be bent into this kit at all. None of that is established for an outsider; it's assumed.

Here is the thing the framing hides. Claude Design already ships a README that tells a coding agent how to implement the bundle. So who is the user of design-handoff-adopt? It is you, one developer, on the maybe-three occasions a year you export a bundle. You are proposing to build, document, regression-test, and maintain a fourth skill whose entire job is to translate an artifact that comes with its own instructions into a house format that exists only because you invented it.

Strip the vocabulary and the question is: "Should I write a permanent tool to reformat a thing that already knows how to be implemented?" Said that plainly, the answer is no.

The honest ceiling confirms it. Token transcription is mechanical (a script, not a skill). Everything with judgment — reconciling multiple aesthetics, pixel-perfect recreation — is exactly what the README already hands to a human-plus-agent. A skill can't make that repeatable; it'll collapse into bespoke judgment every run, which is your pressure-test (1) answered.

Pick D. Do the next bundle as a one-off, write down what was actually mechanical, and only then consider extracting a script — not a skill.

04 The Long View

"Extract, don't generate" only compounds if the kit owns the artifact downstream. Three years out, the binding question isn't "can a skill transcribe 40 tokens" — it's who maintains the design system after adoption. Claude Design will keep shipping bundles; its README will keep improving; its output will get richer and more idiosyncratic every release. A skill whose core job is transcription is racing a moving target you don't control. That depreciates fast.

What compounds is the reconciliation layer — the kit's two-tier tokens.css convention, the write guard, the doc structure. That's yours, it's stable, and it's what makes a bundle survive contact with a real codebase over months of edits. So the durable asset isn't "adopt"; it's "the kit has an opinion about how an external design system gets governed once it's in."

That reframes the build. A fourth sibling deepens the non-composition debt: you'd then have four un-orchestrated nodes, and the orchestrator you haven't built gets harder, not easier, because adopt and bootstrap are mutually exclusive entry modes that must be chosen between — which is exactly a routing decision an orchestrator makes. Option B (a mode in bootstrap) pays that debt down: one skill, two entry modes, one place the orchestrator routes to.

Honest ceiling: transcription automates; multi-aesthetic reconciliation and pixel-perfect recreation never will — those stay human every run, README or not. So scope the skill to the part that compounds (governance + token-tier mapping) and let it defer to the README for recreation.

Opportunity cost: building skill #4 is weeks not spent on the orchestrator, which is the actual bottleneck. Build the mode (B). Defer the standalone.

05 The Veteran

Without auto-composition, this is a fourth standalone node you invoke by hand on a bundle you already have sitting in a folder. That is the part that should worry you.

The standard playbook says: don't build the inverse skill, write the README's procedure into a slash command and run it. Most "ingest an existing system" tools collapse into bespoke judgment on every run, and you end up maintaining a skill that does less than a careful human reading the Claude Design README. Pressure-test (1) is the whole ballgame: "extract/transcribe, don't generate" is only a real skill where it's mechanical — and exactly one part is mechanical. Parsing :root custom properties into the two-tier tokens.css, preserving oklch() values verbatim, wiring class-based theme variants: that is deterministic, repeatable, and genuinely better than freehand because it can't drift or hallucinate a palette. That part earns its keep.

Everything past the token transcription is judgment that does not compress into a procedure. Multi-aesthetic reconciliation is a design call you make per bundle. Pixel-perfect component recreation from prototypes is the exact thing Claude Design's README already hands the implementing agent — a skill wrapping it adds a node, not capability. That is your honest ceiling: reliable token adoption, advisory everything else.

So the obvious move is right with one correction: this isn't a sibling skill, it's a token-adopt command plus a documented manual recreate step. Build the mechanical 20% as real automation under the existing write guard; leave the 80% as guided prose. Don't spin up a fourth peer to the chain you already admit doesn't compose. Verdict: not (A). A scoped command, closest to (B).
