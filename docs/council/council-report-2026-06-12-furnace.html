Plan-Review Furnace: Build Now or Validate First?

5 advisors · anonymous peer review · chairman synthesis

The question

Should Rex build a plan-time "furnace" prompt NOW (a markdown planning prompt seeded with rules R1–R4 targeting plans that assert things they never verified), or run ONE MORE VALIDATION BATCH first to confirm the patterns are stable? Evidence: an 83-session audit of his Claude Code history found ≥57% of plan-review rounds (102 of 178) drew a revision, dominated by a preventable "asserts-what-it-never-verified" class (~40–50 rounds). Known weaknesses: a large share of revisions are external reviews Rex pasted in (not his own catches); the preventable/healthy-friction split is his single unverified judgment call; counts are a floor, not a point estimate. The repo's own philosophy: "earn a guardrail with evidence from the log, not instinct." The furnace is a cheap, deletable markdown prompt.

The recommendation

Council verdict

Write the narrow furnace now — scoped to only the R1–R4 verification-citation core — and let the next cold batch grade it, because a deletable markdown file is the hypothesis, not the irreversible bet your evidence bar is built to guard.

Build the narrow furnace now — siding with the Architect and Veteran — but adopt the peer round's cheaper precursor first. The deciding logic is the cost-structure exception the Veteran named and all five reviewers ratified: your evidence bar exists to stop you embedding expensive, hard-to-remove machinery on a noisy signal. A markdown prompt you can toggle off per session is none of those things. Applying the wait-rule here over-applies judgment until it becomes ritual — exactly the failure mode your own CLAUDE.md warns against. The irreversible fork would be freezing the [P]/[H] boundary and promoting R5–R7, which you should NOT do. Scope the furnace to ONLY the verification-citation discipline: a plan must cite what it read before asserting codebase facts or platform mechanics. That core is contamination-proof — true regardless of which reviewer flagged it — so it clears your evidence bar today on its own merits, independent of the ±5 counts and the unverified split. Drop R5–R7 and the [P]/[H] judgment calls from the prompt entirely. Then build and validation become a single act: run the next ~10 real sessions with the narrow furnace on, count T1/T2/T6 recurrence. The furnace is the falsifiable hypothesis; the cold batch is the test that earns — or kills — the rest.

The one thing to do first

Re-tag the existing 102 rounds by who-caught-it — Rex vs. pasted reviewer — for the T1/T2/T6 "never-verified" class only. Free, no new sessions, and it directly sizes the contamination on the exact core you'd build the furnace from. If those rounds are mostly Rex's own catches (as Reviewer 4 predicts), the contamination objection collapses and you build the narrow furnace immediately; if mostly pasted-in, you've learned the furnace targets a stranger's profile before writing a line of it.

Where the council agrees

Four of five advisors converge on a sharp factual split: the pasted-review contamination poisons the judgment-call rules (R5–R7, the [P]/[H] boundary) but NOT the mechanical verification-citation core (T1/T2/T6). A plan that asserts a file's contents without reading the file is wrong regardless of who caught it — the single most load-bearing point in the deliberation, and it dissolves much of the apparent conflict. There is also broad agreement that the [P]/[H] split is the weakest link, that R5–R7 should not be promoted until cleaner data clears them, and that the cold-session batch is the right experiment to run regardless. Every advisor treats the repo's "earn a guardrail with evidence" rule as genuinely binding.

Where the council clashes

The real clash is build-narrow-now (Architect, Veteran) vs. wait-and-isolate-first (Skeptic, Stranger, Long View). The wait camp's strongest claim is the Long View's: building installs a filter before you've calibrated the lens, permanently contaminating your future signal. The build camp's rebuttal — delivered by the Veteran and ratified by all five peer reviewers — is that this overstates a markdown file you can toggle off in thirty seconds. That near-unanimous verdict effectively settles the clash: the wait camp's central asymmetry (cheap to write, expensive to trust) holds for embedded pipeline guardrails, not for a per-session-toggleable prompt. Reasonable advisors disagreed because they priced different costs — the wait camp the epistemic cost of trusting a tuned-to-noise guardrail, the build camp the reversibility.

Blind spots the council caught

The peer round surfaced gaps every advisor missed. (1) The cold batch has no ground-truth oracle for false-negatives — it measures the precision of Rex's catch profile, never recall: the lies that ship unchallenged because nobody caught them stay invisible. (2) A cheaper discriminating test exists that no advisor proposed: re-tag the existing 102 rounds by who-caught-it (Rex vs. pasted reviewer) before spending any new session — free, and the ~40–50 'never-verified' rounds are most likely the ones Rex caught himself, which would collapse the contamination objection. (3) No one asked whether a plan-time PROMPT can even fix this — the failure is the model not reading files before asserting, an agentic-execution gap a planning prompt may not reach; the batch must instrument whether the furnace changes behavior at all. (4) 'Contamination' may be the wrong frame — pasted reviews catch real defects regardless of source; the question is which catches a plan-time prompt can preempt, not whose catch it was.

Advisor responses

01 The Skeptic

The data isn't measuring what the furnace is supposed to fix. You audited a blended stream where a large share of "revisions" are external reviews Rex pasted in — councils, other Claude sessions. R1–R4 are therefore fitted to what an external reviewer flags, not to what Rex's own plans reliably miss. Build the furnace on that, and you've hardcoded a stranger's review profile into your plan-authoring prompt. That's not a guardrail; it's cargo-culting someone else's instincts. Worse, the one number that decides the whole thing — the [P]/[H] split — is a single unverified judgment call by the person who most wants the furnace to exist. Motivated reasoning lives exactly there. If even a third of your "preventable" rounds are actually healthy human-in-loop friction, the furnace's job becomes suppressing the questions Rex needs his plans to raise. That failure is invisible: a furnace that quietly stops surfacing judgment calls produces confident, wrong-but-unchallenged plans, and you won't see the revisions you no longer get. And the repo's own law convicts you: "earn a guardrail with evidence from the log, not instinct." 32 confidently-promotable rounds out of 102, counts at ±5, a known under-captured channel — that is instinct wearing a spreadsheet's clothes. The 83-session audit produced a floor, not a finding. The asymmetry Rex is underweighting: the furnace is cheap to write but expensive to trust. Once it's in the plan loop, every future plan inherits its blind spots silently. The validation batch — cold sessions, no pasted review — is the cheaper experiment precisely because it tests the load-bearing assumption before it's load-bearing. Validate first. Don't build a guardrail against a signal you haven't isolated.

02 The Architect

Plan-review revisions are not your problem. The waste is real but it's a downstream symptom, and you're about to spend your evidence-quality anxiety on the wrong axis entirely. Strip it down: what are you actually trying to learn? You think the question is "are R1–R4 stable or are they pasted-review noise?" But the furnace doesn't need R1–R4 to be stable. The furnace needs ONE thing to be true: that a plan which is forced to cite what it verified produces fewer "asserts-what-it-never-checked" rounds. That's T1+T2+T6 — and you already named it as a single mechanical class, ~40–50 rounds, independent of which external reviewer flagged it. A plan that reads the file before asserting the file's contents is correct whether Rex, a council, or a random Claude caught the lie. The pasted-review contamination poisons R5–R7 (the judgment-call rules). It does NOT poison the verification-citation core. So you're optimizing the wrong variable. "Confirm patterns are stable" is a research instinct; the furnace is a cheap, reversible markdown file. You don't validate a hypothesis you can A/B test for free. Smallest move that actually works: write the furnace NOW, but scope it to ONLY the verification-citation discipline (R1–R4's epistemics, drop the [P]/[H] judgment calls entirely). Then your "validation batch" and your "build" become the same act — run the next 10 real sessions with the furnace on, count T1/T2/T6 recurrence. That's the cold-session experiment you wanted, except it also ships the fix. Waiting buys you a cleaner number about a thing you can just measure live. Build the narrow furnace. Let the next batch grade it.

03 The Stranger

The data isn't measuring what the furnace would fix. The audit mines "plan-review rounds," but a large share of those rounds are external reviews Rex pasted in — councils, other Claude conversations. So R1–R4 aren't a profile of "my plans miss X"; they're a profile of "things some reviewer somewhere flagged." A plan-time prompt seeded with that signal is trained on a population it will never face at plan-authoring time. That's not noise you can average out with more sub-agents — it's the wrong unit of analysis. Three buried assumptions an outsider would stop on. "Furnace" is doing heavy lifting — it's never defined as a mechanism, only as "a markdown planning prompt seeded with rules." If the artifact is genuinely that cheap, the entire build-vs-wait framing is suspicious: you don't convene a deliberation over writing one prompt file you can rewrite tomorrow. Either it's cheap (then just build it and let reality grade it) or it's not (then say what the real cost is). Second, the [P]/[H] split is described as load-bearing and "his judgment call, made once, unverified" — the whole furnace targets a category boundary nobody has tested. Who's affected: one developer, future-Rex, in cold planning sessions. He's the entire population. So the cleanest validation isn't "re-mine a fresh batch" — that just re-reads the contaminated log. It's the cold-session probe he already named: run plans without external review and watch what HE actually misses. That directly tests the contamination hypothesis instead of laundering it. Verdict: run the cold-session batch — not to confirm patterns, but to separate his catch profile from the pasted reviewers'. Build nothing until that separation exists.

04 The Long View

Building now is the cheaper-looking mistake. The furnace is "cheap to write," yes — but its real cost isn't the markdown, it's what it does to your signal for the next three years. Once a furnace exists, every future session runs through it, and you stop seeing the raw revision stream cleanly. You've installed a filter before you've calibrated the lens. From then on, you can't easily tell whether revisions dropped because the furnace works or because it's suppressing the [H] friction you explicitly said you want to keep. The instrument contaminates the measurement it was built from. That's the second-order effect you're not pricing. Run the cold-session batch first — but understand what compounds there. The asset that appreciates isn't R1–R4. It's a clean Rex baseline: a measurement of your own plan-authoring catch profile, uncontaminated by pasted external reviewers. That baseline is reusable forever. It tells you which guardrails are yours versus which are council-shaped. Every future furnace, rule, and skill you build gets validated against it. R1–R4 tuned to a blended signal depreciate the moment your reviewer mix changes (different council, different model). The baseline doesn't. Your own repo philosophy — "earn a guardrail with evidence from the log" — is the reputation asset here. 32 confidently-promotable rounds against a stated evidence bar is you about to violate your own constitution on instinct. Do that once and the bar means less every subsequent time you invoke it. Protect the principle; it compounds harder than any single furnace. Wait. Run the batch. Build the furnace from clean Rex.

05 The Veteran

The obvious move here is the standard playbook, and it says wait. "Earn a guardrail with evidence from the log, not instinct" is Rex's own rule, and 32 confidently-promotable rounds out of 102 doesn't clear it. I've watched a dozen people build the controller before the measurement was stable and then spend three cycles untangling whether the tool or the noise moved the metric. That's the pattern, and it usually wins. But this case is genuinely different, and the difference is the cost structure. The furnace is a markdown prompt. Building it is not the irreversible bet the playbook is built to protect against — tuning a guardrail to noise only hurts when the guardrail is expensive to write, expensive to rip out, or silently embedded in a pipeline. None of that is true here. You can delete it in thirty seconds. So the real question isn't build-vs-wait. It's: what's the cheapest thing that produces a clean signal? And Rex already named it — cold sessions without pasted external review isolate his own catch profile. That's the actual experiment. The mistake would be running the validation batch as a substitute for building, then having nothing to test against. Do both, in the right order: write a thin furnace seeded with only R1–R4 — the T1/T2/T6 "asserts what it never verified" core, which is the one finding that doesn't depend on the [P]/[H] split being right. Then run the cold batch as the test of that artifact. The furnace becomes the hypothesis the batch falsifies. Don't promote R5–R7 or freeze the [P]/[H] boundary until the cold data clears them. Build narrow now; let the batch earn the rest.
