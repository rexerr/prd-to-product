Council Report

Self-Healing Plan Review Loop — build it, or earn it first?

5 advisors · anonymous peer review · chairman synthesis

The question

Should Rex (a solo developer) build a "Self-Healing Plan Review Loop" — a system that captures the feedback he gives when reviewing Claude Code's implementation plans, tags it by theme, counts it, and fires a STOP gate when a tag crosses a threshold (default 10) so he runs a human "improvement sprint" that folds the fix back into Claude Code's context so the same feedback stops recurring? It would live as a parallel docs/plan-reviews/ subsystem in a repo already meta-system-dense whose core principle is "earn a guardrail with evidence from the log, not instinct." Claude counter-proposed a lighter path: fold tagged feedback into existing session retros, grep as the counter, build the machinery only after a tag trips by hand. Rex also wants to trial the system on a different live project (his llm-council tooling) before committing it to the home repo. The council was asked: is the mechanism sound; is the volume critique decisive; is fold-into-retros the right call; is the separate-project trial smart de-risking or signal-dilution; what's the single highest-leverage first move.

The recommendation

Council verdict

Build nothing — answer "what do I re-correct?" from memory today; only if you genuinely can't should you append a tagged line to existing retros for two weeks, because the subsystem (and arguably the trial too) fails your own "evidence not instinct" rule at solo volume.

Skip data collection. Answer from memory first. The peer reviews are right that the two-week trial reproduces the original sin and risks an uninterpretable n. So do the cheapest possible falsifying test, which costs ten minutes and no infrastructure: sit down and list the plan-review corrections you re-give. Be honest about whether each is a stable, promotable pattern (e.g. "always reads the wrong sibling file," "never checks scope limits before coding") versus a one-off project-specific correction.

Three outcomes:

1. You can name 2-3 stable, recurring, promotable patterns. Then the premise is true and the system is unnecessary — fix them directly. Fold those fixes into CLAUDE.md or, per the Architect's genuinely good point, into a single planning-prompt.md / plan-review checklist the drafter reads. Build that artifact (the furnace), because it's useful day one regardless of any loop. Build no counter, no ledger, no script, no STOP gate.
2. You can't name them, or they're all one-off. Then the recurring-feedback problem is a phantom at your scale. Build nothing and drop the proposal.
3. You're genuinely unsure and want the count anyway. Only then append one tagged line to existing retros — never a parallel docs/plan-reviews/ tree — and pre-commit now to the kill/build numbers (e.g. "if no single tag reaches 5 in a month, kill it"). Without a pre-declared threshold the count is theater.

On the separate-project trial: don't. Unanimous and correct — test where the feedback lives or not at all. The llm-council trial dilutes signal and defers the same null result against a different baseline.

The Architect's furnace is the one constructive idea worth keeping — but gated behind the memory check, not ahead of it. Don't wire a thermostat into a room you're sitting in with your hand already on the heater.

The one thing to do first

Spend ten minutes writing down the recurring plan-review corrections you give, from memory, and mark each as "stable/promotable" or "one-off." If you can name 2-3 stable ones, fix them directly (fold into CLAUDE.md or a new planning-prompt.md) and build no machinery. If you can't, the system is solving a phantom — drop it.

Where the council agrees

All five advisors converge, and the peer reviews ratify it (The Skeptic won "strongest" 5/5):

• Do not build the parallel subsystem now. Unanimous. The entries/ + LEDGER.md + sprints/ + review-pulse machinery is premature.
• The volume critique is decisive. At 5-15 events/week across 8 tags, a threshold of 10 trips only after weeks or months — by which point the count measures filing patience, not stable recurrence. This isn't Toyota's line.
• The kill shot is the missing count. You have zero evidence you re-fix the same things. If it's true, you can name the top-3 now; if you can't, the machinery solves a phantom.
• Back-propagation has no target. The "teeth" fold fixes into a planning-prompt artifact that doesn't exist. The loop is toothless until that surface exists.
• The separate-project (llm-council) trial is signal-dilution, not de-risking — different repo, different failure modes, no shared baseline.

• The taxonomy / WHERE-it-lives tag is the one cheap, real salvage.

Where the council clashes

Largely unanimous on the verdict. One genuine fault line on the first constructive move:

• The Architect (furnace-first): the binding constraint isn't capture, it's that no durable planning surface exists, so every plan re-earns the same corrections from CLAUDE.md cold. Build the planning-prompt.md / plan-review checklist now — useful day one, independent of any loop, and where 80% of recurring feedback would land.
• The other four (count-first / build-nothing): don't build any artifact yet; collect the count (or recall it from memory) and let evidence decide. The Architect green-lights infrastructure on the same instinct the repo forbids — it assumes the recurring-feedback problem is real before it's counted.

This is the only real disagreement, and the peer reviews flag the Architect as the chief blind spot precisely here.

Blind spots the council caught

Peer review surfaced gaps that cut underneath the unanimous "just collect a count" prescription — the sharpest contribution of the round:

• The retro-tagging trial is itself an instinct-adopted guardrail. All five prescribe "tag for two weeks then count," but none asks whether that mini-loop reproduces the proposal's original sin at smaller scale.
• The count may be uninterpretable at this volume. n is too small to distinguish recurrence from noise; no one specifies what count justifies building vs. kills the idea. An evidence step that yields an uninterpretable result is false precision dressed as rigor.
• The feedback may not be recurrence-shaped at all. Much plan-review feedback is project-specific and one-off — not a CLAUDE.md-promotable pattern. If so, no threshold ever fires legitimately, and even a tripped tag may be unpromotable.
• Measuring can manufacture the signal. Tagging every review trains attention onto the friction and can create the recurrence it claims to neutrally observe. And the counter measures throughput of annoyance, not whether the annoyance wasted time.
• Tagging adds ceremony to a workflow whose complaint is ceremony. "Append one tagged line" is treated as free; it's per-review friction added to the exact process being optimized.

Advisor responses

01 The Skeptic

The mechanism has a flaw the counter-proposal undersells: at 5-15 items/week with 8 tags, you are dividing roughly 10 events across 8 bins. A tag hits 10 only after weeks. By then the "recurrence" you detect is contaminated by every CLAUDE.md edit, skill change, and mood shift in between — you cannot attribute the trip to a stable pattern versus noise. This isn't a thermostat; it's a thermostat reading a room where someone keeps moving the thermometer. The control-loop framing is doing persuasive work the statistics don't support.

Worse: the system's own success destroys its signal. Promote a rule, reset the counter — now that class can't re-trip for weeks even if your fix was wrong. You've built an instrument that goes quiet exactly when you'd want to know the fix failed.

The central irony is the kill shot and it's correct: you have zero count proving "I fix the same things repeatedly." That claim is the entire load-bearing premise, and it's instinct. If it were true, you could name the top-3 recurring problems right now without any system — so name them. If you can't, the premise is false and the machinery is solving a phantom.

The hidden-dependency point compounds this: back-propagation needs a planning-prompt artifact that doesn't exist. The "teeth" are vaporware.

The separate-project trial is signal-dilution, not de-risking — different repo, different failure modes, no baseline; you'd be testing whether the machine runs, not whether it helps.

Highest-leverage first move: spend ten minutes listing your actual recurring plan-review failures from memory. If the list is short and obvious, fix those directly and build nothing. That test costs nothing and can kill the whole project.

02 The Architect

Build nothing yet. You are optimizing the wrong variable.

The variable you think you're optimizing is "feedback recurrence." The variable that actually determines whether any of this pays off is "does a planning-prompt artifact exist to fold fixes into?" Back-propagation is the teeth, and the teeth bite into an artifact that doesn't exist. You're designing a thermostat with no furnace wired to it. The counter, the tags, the STOP banner — all of it actuates through a promotion target that's currently vapor.

So reframe: the problem isn't "capture feedback." It's "I have no durable planning surface, so every plan starts from CLAUDE.md cold and re-earns the same corrections." Build the furnace first. Create a single planning-prompt.md (or a plan-review checklist the drafter reads). That's the missing artifact, it's useful on day one independent of any loop, and it's where 80% of recurring feedback would actually land.

The simplest first move that actually works: for the next ~15 plan reviews, append one tagged line to your existing retro. No new directory, no script, no banner. Grep it. That costs an hour total and produces the count the whole proposal currently lacks. You either see a tag cluster — which earns the machinery with evidence, exactly your repo's principle — or you don't, and you've spent an hour instead of a weekend building a thermostat for a room that never gets cold.

On the trial: testing on llm-council first is signal-dilution, not de-risking. The recurrence pattern is specific to this repo's CLAUDE.md. Test it where the feedback actually lives.

03 The Stranger

The volume critique is decisive, and here is the angle nobody inside this repo can see: the proposal is a recursion. Rex is using his plan-review loop to design a system that captures lessons from his plan-review loop. The first artifact he should produce is not a taxonomy. It is one honest count of how many times in the last month he actually re-typed the same feedback. He has not produced it. That absence is the whole verdict.

Words that wouldn't survive outside Rex's head: "thermostat for context engineering," "back-propagate," "actuator," "STOP banner," "improvement sprint." These are control-systems metaphors dressed as a build spec. A thermostat is worth building because a house has thousands of temperature events a day and no human watching. Rex generates 5-15 events a week and is personally watching every one. You do not install a thermostat in a room you are already sitting in with your hand on the heater.

What Rex assumes the reader already wants: a system. The unstated belief is that a recurring annoyance deserves infrastructure. But the affected party here is one person, and what that person will actually experience is: writing markdown entries, maintaining 8 drifting tags, and glancing at a counter that says "3 of 10" for two months before it ever fires. That is not relief from repetition. That is a second loop of work bolted onto the first.

Who is affected: only Rex, and future-Rex inherits the maintenance.

Highest-leverage first move: skip Phase 0's machinery. For two weeks, append one tagged line to your existing retro each time you re-give feedback. Then count. If a real number forces your hand, build then. If memory already fixed it, you saved the whole thing.

04 The Long View

The repo's principle is "earn a guardrail with evidence not instinct." The most expensive thing you own here is not the plan-review loop — it's that principle's credibility. Build this subsystem on instinct and you've spent the one rule that makes every other rule in this dense meta-system trustworthy. Three years out, the question isn't "did the thermostat reduce recurrence" — it's "is the log still load-bearing, or did it become a place where ceremony goes to look like rigor." You can't measure the second one, and that's exactly why it's the one that compounds.

Now the asset that actually compounds: this thing is a product candidate. That reframes everything. An internal tool fails quietly; a scaffolded skill that ships premature ceremony into every future project fails in every clone, and you inherit the maintenance of a taxonomy that drifts under eight tags at a volume where it can't earn its keep. The depreciating asset is the markdown subsystem you'd build now. The compounding asset is the evidence that the mechanism works at solo throughput — and you have zero of that today.

So the highest-leverage first move is not Phase 0. It's the cheapest possible instrument that produces the missing count: add the plan-review feedback line to existing retros, grep weekly, and do nothing else for a month. If a tag genuinely trips by hand, you've earned the subsystem with the exact evidence your own principle demands — and you'll have proof it's worth productizing. If it doesn't trip, you've saved yourself from shipping a guardrail-shaped artifact into every future repo.

On the separate-project trial: that's signal-dilution. Trial it where the workflow actually hurts.

05 The Veteran

The obvious move here is the standard playbook: when feedback evaporates and you repeat mistakes, build a capture-and-promote loop. I have watched a hundred solo developers build exactly this. The pattern is reliable, and so is its failure mode: the meta-system outlives the problem it was built for. You build the andon cord, honor it twice, and six months later you're maintaining scripts/review-pulse and reconciling tag drift against a problem you'd have solved by just remembering.

Claude's volume critique is the decisive one, and it's not close. Andon cords, Kaizen counters, CI-for-prompts — every one of those patterns earns its keep at a volume where human memory provably fails. Toyota's line throws thousands of events. You throw 5-15 a week. At that volume the counter isn't measuring recurrence, it's measuring your patience for filing entries. The machinery is load-bearing at 500 events/week and pure ceremony at 10.

And this repo already told you the answer. "Earn a guardrail with evidence from the log, not instinct." The proposal violates the house rule on the first move — it asserts "you fix the same things over and over" with zero count. That's not irony, that's the rule working. Honor it.

What's genuinely special here, the one thing that survives: the taxonomy and the WHERE-it-lives tag. That's real and cheap. Adopt Claude's lighter path — a plan-review feedback section in existing retros, one tagged line per item, grep as the counter. Build the parallel subsystem only when grep shows a tag tripping by hand.

The separate-project trial is signal-dilution. Highest-leverage first move: add the tagged line to retros for two weeks. Let the log earn the system — or kill it.
