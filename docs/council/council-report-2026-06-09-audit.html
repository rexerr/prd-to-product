Council Report

Final audit: adopt Claude Design tokens — build, defer, or reference?

5 advisors · anonymous peer review · chairman synthesis

The question

FINAL AUDIT, with fidelity evidence in hand. A single-developer kit of single-purpose, markdown-only skills (prd-creator, context-engineering, design-system-bootstrap=DSB) scaffolds AI-coding projects. The user designs in Claude Design, which exports handoff bundles (HTML/CSS/JS prototypes + an existing CSS-custom-property design system + screenshots + a PRD + a README telling agents how to implement). A fidelity test transcribing the real bundle into the kit's two-tier tokens.css returned VALUE-LOSSLESS (all 40 tokens copy verbatim) but STRUCTURALLY NON-ISOMORPHIC (DSB's scale-first template fabricates shades; 3 vs 6 fonts; no tier for 5 categorical accents / a --crt scalar / a toggle; one dark block vs base+twilight+3 surfaces+N colorways). Theming is mechanical. Refined plan to audit: no 4th skill, no DSB mode; IF anything, a small token-adopt slash command emitting flat-primitives + thin alias layer + N theme blocks; recreation delegated to README. Build now, defer until a 2nd bundle, or don't transcribe at all (keep bundle authoritative, reference it)?

The recommendation

Council verdict

Don't build the command — but don't pretend 'reference the bundle' is free either: ship the written fidelity baseline now and copy the tokens into the product repo as a one-time cp, because the prototype is disposable and the app has to build without it.

Hold the unanimous verdict where it's earned and override it where the reviews caught it overreaching. KEEP KILLED: the 4th skill and the DSB mode — dead, correctly. DON'T BUILD THE SLASH COMMAND YET, but for the right reason: not because it's a dangerous importer (it isn't — it's a saved prompt against a static bundle you own), but because at n=1 you can't tell a theme from tweak-tool scratch, so you'd hardcode an N and a schema (--crt, the toggle, 5 categorical accents) you can't validate; the command is cheap to write later, which is exactly why deferring it costs nothing. BUT THE PRODUCT STILL NEEDS TOKENS NOW — the gap the council missed and the part that is not deferrable. The 40 lossless values must land in the real app's repo to build, so the correct move is the dumbest one: a one-time cp of the bundle's tokens into the product repo as its canonical tokens.css. Not a symlink (breaks when the prototype is discarded), not an @import (same), not 'reference in place' (fiction) — a literal copy. The bundle stays authoritative as the source of record for component recreation (its README does that job); the product repo owns its built tokens outright. Lossless values are what make a one-time copy safe — nothing to transform, nothing to rot. WRITE THE RECIPE NOW as a recorded baseline, not a reusable tool: the fidelity finding plus the manual transcription steps go into one page in the kit's docs, framed as 'what this one bundle required and why it didn't fit DSB's schema' — an observation log, not a general procedure — which neutralizes the n=1-in-prose risk and gives bundle #2 a recorded diff target. Net: kill two, defer one (cheaply), ship two now (the one-time copy into the product repo, and the recorded fidelity baseline).

The one thing to do first

cp the bundle's tokens into the product repo as its own canonical tokens.css (one-time, no symlink/import), and in the same commit drop a one-page fidelity baseline in the kit's docs recording what this bundle required and where DSB's schema fabricated shades — so the product builds today and bundle #2 has a recorded diff target.

Where the council agrees

Unanimous and correct on the negatives: no 4th skill, no DSB mode, don't bend DSB's scale-first template to a shape it fabricates. All five also agree the bundle is effectively self-transcribing — lossless values plus a README mean a transcriber buys almost nothing over the agent reading the file directly. And all five land on 'defer / build nothing now,' with the command (if ever) gated on a second real bundle.

Where the council clashes

Two real splits. First, on whether the threat is even live: Long View, Veteran, and the Architect treat the artifact as tracking an upstream you don't control (de-sync, depreciation, importer-rot) — but a peer review catches that the bundle is a static one-off the dev OWNS, so the 'moving target' framing is overstated for this artifact. Second, on what 'defer' costs: the Skeptic treats undefined N as a reason to build nothing, while a reviewer flips it — resolving N (theme vs tweak-tool scratch) is exactly the human judgment a recipe should capture now, not wave away.

Blind spots the council caught

The peer reviews exposed three things all five advisors collapsed. (1) Kit vs. product: every advisor said 'keep the bundle authoritative, reference it,' but the bundle is a throwaway prototype export — the real app needs the tokens IN ITS OWN REPO to build. 'Reference in place' is right for the kit and a fiction for the product; when the prototype is discarded, an in-place reference points at a file that no longer exists, and none of the advisors said where canonical tokens live post-disposal. (2) Command vs. saved prompt: all five inherited the 'importer rots into a maintenance tax' failure wholesale, but a ~30-line command you re-run on demand against a bundle you're holding is a saved prompt, not a trusted generator — near-zero standing maintenance, cheap to write, cheap to delete; 'defer until proven twice' over-weights a lock-in risk that barely exists for a disposable re-runnable artifact. (3) The fallback isn't free and the mechanism is unspecified: the endorsed manual recipe carries the same n=1 schema-fitting risk they condemn in the command, and 'reference, don't transcribe' never names a mechanism (symlink? @import? a pointer in CLAUDE.md?), so the actual deliverable stays unbuilt.

Advisor responses

01 The Skeptic

The plan is correct in its negatives — no 4th skill, no DSB mode. The fight is the residual "build a small command." N=1 — one bundle, one designer, one tool version; the test names the instability (4 colorways may be tweak-tool scratch, not V1 themes), so the format is undefined at the boundary that matters (theme vs scratch) and a command emitting "N theme blocks" must decide an N you don't know. Deeper: you're building a transcriber for an artifact that ships its own transcription instructions — a README plus a working var() alias. Values are lossless precisely because they're already valid CSS in a file, so the command only saves a copy-paste the agent does anyway. The fidelity test accidentally argues the last option: keep the bundle authoritative, reference it. Failure mode: Claude Design changes its export shape (it will), and your hand-extracted command is silently wrong until a project breaks. Verdict: right on no-skill/no-mode; wrong instinct on the command. Build nothing now.

02 The Architect

DEFER — but not toward "build now." What are you actually solving? Not "adopt tokens" — getting the bundle implemented in a real project with the designer's values intact. The test proved values copy verbatim and the bundle ships its own README; the authoritative artifact already exists and is lossless. A transcription step is net-negative until proven: a second copy that can drift, and you'd hand-author a NEW schema (flat-primitives + alias + N theme-blocks) off n=1. The real risk is schema lock-in on n=1 — the next bundle might have scales, 2 fonts, or no colorways; codifying now bakes today's idiosyncrasies (--crt, --placeholder-tag-display) into a "format." Simplest move that works: do the next bundle BY HAND, keep the bundle authoritative (reference, don't re-home), capture the manual steps as a short docs checklist, and extract a command only when bundle #2 confirms the flat-primitive shape across two real inputs. The Rule of Two applies exactly. Right to kill the 4th skill and the DSB mode; wrong to build the command now.

03 The Stranger

"Value-lossless but structurally non-isomorphic" assumes lossless-values is the win — but you proved the opposite. If all 40 tokens copy verbatim, transcription is cp. The value of a skill is the structure it imposes — and the structure is exactly what doesn't fit. You ran a test that confirmed the mechanical part is trivial and the judgment part is hard, then proposed building a tool for the mechanical part. That's backwards. "IF anything is built" — notice the author already wants to build something. A single developer, by the time a second bundle exists, will have forgotten the command's assumptions and re-derive the shape by hand anyway — faster than debugging a stale generator against a format that "N colorways" already says is unstable. The bundle ships its own README — the authoritative artifact already exists; a command competes with it and drifts. Keep the bundle authoritative; reference it. The --crt scalar and --placeholder-tag-display toggle are the tell — bespoke escape hatches mean the next bundle has different ones. Write the transcription as a documented manual recipe now; build nothing until the format proves itself twice.

04 The Long View

The fidelity test didn't just measure fit. It told you the bundle's shape is the UPSTREAM's, and the upstream is a moving target — the load-bearing three-year fact, arguing harder than the plan admits toward "don't transcribe." What compounds is not a token file — that depreciates the instant Claude Design ships a new export schema (and it will; it's a young product iterating fast). What compounds is the CONVENTION that the bundle stays authoritative and your kit references it — that survives every upstream format change for free. A transcribe command does not: each schema revision silently de-syncs your emitted file from the source of truth, a maintenance liability that tracks a release cadence you don't control. Building a command to a shape you've seen once is fitting a schema to n=1. DEFER is correct, but be honest that "defer until a 2nd bundle" is likely "defer indefinitely" — and that's the right outcome, not a delay. A 4th surface, even a small command, dilutes the kit's strongest asset (single-purpose legibility) and is the first one whose correctness you don't own. Three-year failure mode: Claude Design adds an importable tokens.css and your transcriber is strictly worse than a symlink. Defer the command; ship a one-page reference convention now.

05 The Veteran

The pattern: "ingest an external format" tools die one of two ways — either the upstream format drifts and your importer rots into a maintenance tax, or the importer subtly lies, fabricating what the source didn't specify. Your fidelity test already caught the second failure mode in the act: DSB's scale tier fabricates shades. That's the canonical importer sin, found before shipping. So the refined plan is mostly right, for the standard reason: don't build the 4th skill, don't bend DSB. Two corrections. First, the open sub-question answers itself — N=1; every importer built against a single sample encodes that sample's accidents as the format (the --crt scalar, the 5 categorical accents, the twilight-plus-3-surfaces shape). Defer; build it on the 2nd bundle, not the 1st — that's the only way you learn which fields are the format. Second, this case may be special: if the bundle ships its own README and recreation already delegates there, the bundle is already authoritative; transcription buys you a tokens.css the agent could read directly. Ask what emitting a file gets you that referencing the bundle in-place doesn't — if "a write-guarded canonical location," that's a thin win for a new surface to maintain. Verdict: defer, and pressure-test "reference, don't transcribe" first.
