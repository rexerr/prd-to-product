Council Report

Batch B — port the one-board kanban model into the context-engineering scaffold?

5 advisors · anonymous peer review · chairman synthesis

The question

Should the context-engineering skill's scaffold templates be changed so every newly-scaffolded project is BORN with a one-board kanban work-tracking model (one row per unit; lanes active/next/watching/backlog/blocked/icebox; a Seq ordering column; board-is-the-roadmap; thin-on-touch; retire-don't-re-accrete; detail in tickets/<slug>.md) — instead of the older multi-section backlog shape (Build plan / In progress / Backlog / Open decisions / Later / Done) the scaffold ships today? Graduation machinery already exists (when the backlog outgrows the session-start read, split into a thin index + docs/tickets/ + a retirement ritual); the mismatch between today's shape and that graduation target is a soft seam, not a hard contradiction. The model is proven in the meta-repo that builds the skill but unproven on arbitrary scaffolded projects. The decision is costly + hard to reverse (bakes structure into every future project; changing later means a brownfield migration the workspace avoids). Options: (A) full port now; (B) keep the simple multi-section default; (C) reduced/optional form — lanes-no-Seq, or ship the one-board model as the target the existing graduation rule migrates into, not the day-one default.

The recommendation

Council verdict

Keep the simple multi-section backlog as the day-one default and rewrite the existing graduation rule to migrate into the one-board kanban model — but verify first that real scaffolded projects ever actually trigger graduation, because the whole compromise dies if they don't.

Go with Option C, the 'adopt at graduation' variant, day-one default stays simple — but treat it as provisional on a premise check. Concretely: (1) Day-one default keeps Option B's simple multi-section backlog, Seq-free. This is the robust, low-regret call regardless of how the premise checks resolve. (2) Rewrite the graduation rule's target from 'thin index + generic-status tickets' to 'thin index + one-board kanban (named lanes) + tickets,' each lane citing its failure mode, and point the generic `status` field at the named lanes to close the soft seam. This is the Architect's move, the cheapest coherent step, and it has no day-one coupling — so it's reversible. (3) On Seq: leave it out of the day-one default entirely; include it in the graduation target only if it earns its failure-mode citation. Don't ship Seq as a birthright under any option. I'm siding with the four-advisor consensus over the Long View's pure inaction because repointing the graduation target costs almost nothing, is reversible, and stops the proven shape from rotting as tribal knowledge — the Long View's own migration-liability argument doesn't distinguish its pick from C, since C-at-graduation incurs no installed-base coupling. The one place I override the council's confidence: they all built on an unchecked premise. So the rewrite should not ship until the premise check comes back. If graduation never fires on real projects, the honest move is Option B unadorned — don't dress up a dead-letter rule as a compromise.

The one thing to do first

Verify the two load-bearing premises before writing any rule change. Specifically, pull whatever evidence exists about real scaffolded projects — even informally — and answer two questions: (1) what is the actual horizon/size distribution of projects this scaffold has been used on, and (2) has the graduation trigger ever fired on any project other than the meta-repo? If there is no downstream data at all, that itself is the finding: ship nothing structural now, keep Option B, and add a single lightweight observation hook (a note in the scaffold or a check-in) so the next few real projects generate the signal the council was forced to assume. Only after that signal exists do you rewrite the graduation target.

Where the council agrees

Near-unanimous on the core call: reject Option A (full day-one port), all five advisors. The shared reasoning: the meta-repo is the maximum-favorable case (solo dev, multi-month horizon, work-is-tickets, an author who invented and personally runs the model), not the median scaffolded project — so 'it works here' doesn't transfer. Day-one lanes/Seq/retirement rituals are the over-engineer-on-day-one anti-pattern the project already guards against, and they fail the project's own 'every rule cites its failure mode' test (a three-week app has no roadmap for Seq to order). The bloat problem A claims to solve is already handled by existing graduation machinery — the seam is soft, not a hard contradiction. The one-board model is the right thing to graduate INTO, not to be born with. Four of five converge on some form of C-at-graduation; the Skeptic lands functionally adjacent (ship it as the documented graduation target). The Architect's reframe is the runaway peer favorite (5/5 'strongest' votes): the fork is not which template but when a project pays the cost of lanes/Seq/tickets — and the cheapest move is to repoint the graduation target at the named lanes, each citing its failure mode.

Where the council clashes

This is not a B-vs-C fight in spirit — it's largely unanimous against A and toward 'simple now, kanban at graduation.' The one real axis is how much to ship into the graduation target, and whether to do anything at all now. The Long View is the lone outlier toward pure inaction (Option B, 'wait for a real signal of multiple projects reinventing lanes'). Every other advisor and all five reviewers mark this as the council's biggest blind spot: it leaves the graduation target half-defined and strands a proven shape as tribal knowledge, when repointing the target is a reversible, no-day-one-coupling move. On Seq specifically: the Veteran and Skeptic want lanes-maybe but Seq/roadmap-collapse/accretion-rules explicitly out (expensive, opinionated); the Architect would port the fuller lanes+Seq shape but only at the graduation step. Minor, and resolvable by keeping the day-one default Seq-free regardless.

Blind spots the council caught

The peer reviews are unusually aligned and land on two unverified, load-bearing premises that EVERY advisor assumed without evidence. (1) The median-project distribution is pure assumption. All five built their case on 'the median user is a solo dev shipping a weekend app that never outgrows a flat backlog.' There is zero adoption data, telemetry, or even one named real downstream project behind this. If the skill in fact serves mostly longer-horizon products, the anti-pattern framing collapses and Option A gets materially stronger. One unproven persona is doing ALL the work of defeating A. (2) Whether the graduation trigger ever fires in the wild is also unproven. The graduation machinery is validated only in the meta-repo. If real scaffolded projects get abandoned before their backlog outgrows the session-start read, then 'adopt at graduation' ships a model that arrives never — making the C-at-graduation consensus functionally Option B in a compromise costume. (3) A secondary catch: even the graduation target may export an operator-dependent practice — the retirement ritual works here because Rex runs it; an arbitrary project may never run it regardless of when it's introduced. Given both premises under the consensus are unverified, confidence in the specific shape of the recommendation is moderate, not high. The direction (don't port to day-one) is robust; the value of the graduation-target rewrite is contingent on premises nobody checked.

Advisor responses

01 The Skeptic

Don't be seduced by "it works HERE." The meta-repo is the single most atypical user of this scaffold: single developer, multi-month horizon, the work *is* discrete process-tickets, and the operator personally invented the one-board model and wants it. That is the maximum-favorable case. The scaffold's job is to serve the median case — a solo dev shipping a weekend app — and there the kanban model is overhead theater: lanes, Seq, and a separate `tickets/<slug>.md` card per row is a project-management ritual for someone who has nine TODOs and a deploy deadline.

The fatal flaw in Option A is the failure-mode citation rule. Every emitted rule must name the failure it prevents. What failure does day-one Seq prevent for a three-week project? "Roadmap drift"? There's no roadmap yet. You'd be shipping a rule whose justification is "the workspace that built me likes it" — which is exactly the don't-import-internal-vocabulary line (D-013/D-019) you already drew. Lanes/Seq/retire-don't-re-accrete is workspace machinery wearing a scaffold costume.

And the premise undercuts itself: you state the graduation machinery *already* handles bloat, and the seam to the kanban target is *soft, not a hard contradiction*. So the problem A solves doesn't exist. You'd take on a hard-to-reverse, brownfield-creating commitment to fix a non-bug.

Verdict: A is over-engineering the day-one experience — the exact anti-pattern this project names. Reject A outright. Reject C too if "C" means lanes-on-day-one-minus-Seq; that's the same costume, fewer buttons. The only defensible C is *ship the one-board model as the documented graduation target* — adopted when a project actually outgrows the flat backlog, never on day one. Default stays B. Make the kanban the upgrade you grow into, not the cage you're born in.

02 The Architect

Port a reduced, opt-in form — Option C, and specifically the "adopt at graduation" variant. Here is the reframe.

You are not choosing between two work-tracking shapes. You are choosing *when* a project pays the cognitive cost of lanes/Seq/tickets. The fragmented template and the one-board model aren't competing structures — they're the same project at two different sizes. The real variable is **the trigger**, not the template.

That reframe collapses the fork. A solo weekend app with nine items does not need lanes, a Seq column, or a `tickets/` split — it needs a list. Forcing the kanban on day one optimizes for the meta-repo's problem (a long-lived registry that blew its read cap) and imports it into projects that will never have that problem. That is exactly the day-one over-engineering this project already names as an anti-pattern. Option A loses.

But Option B is too passive — it leaves the one-board model as tribal knowledge the workspace proved out and then declined to ship. The graduation machinery is *already scaffolded* and *already triggered by the real failure mode* (board outgrows the session-start read). That is the seam. The one-board model is what graduation should graduate *into*.

Smallest first move that actually works: keep the fragmented template as the day-one default, and rewrite the graduation rule's target from "thin index + generic-status tickets" to "thin index + one-board kanban (lanes/Seq) + tickets." Add the lane vocabulary to the rule template *at the graduation step only*, each lane citing its failure mode. Close the soft seam you already found — point generic `status` at the named lanes — so graduation lands somewhere coherent instead of half-defined. You ship the proven model exactly where the proven failure mode justifies it, and nowhere it doesn't.

03 The Stranger

Day-one default is the wrong place for this. Here is what someone outside this workspace hears when handed a freshly scaffolded project that is "born" with a one-board kanban: six lanes (active/next/watching/backlog/blocked/icebox), a Seq column they must keep ordered, a "thin-on-touch" rule they can violate, a "retire-don't-re-accrete" ritual, and a tickets/ directory — before they have written a single line of code or have a single unit of work to track. That is a process for a mature, multi-stream repo. The solo weekend-app dev opens BACKLOG.md, sees a governance apparatus, and either fights it or ignores it. Neither is the welcome you want.

The question assumes the reader already wants the meta-repo's discipline. They don't know they have a backlog problem yet — and most never will, because the scaffold already ships graduation machinery that triggers *exactly when* the backlog outgrows the session-start read. That is the tell: you have already built the thing that introduces this model at the right moment. The right moment is not birth.

Who is actually affected: every future scaffolded project, most of which are smaller than this one. They experience the full model as overhead with no felt payoff, because the payoff (a board that scales) only appears at a scale they may never reach.

So: not (A). Lean (C), and specifically the "documented upgrade adopted at graduation" variant — make the one-board model the *target* the existing graduation rule migrates toward, not the day-one shape. That closes the soft seam the question already flagged (generic `status` → named lanes) at the one moment a project has earned the complexity, and it honors the constraint you stated yourself: writes shape, not content; over-engineering day-one for a solo dev is a known anti-pattern. You are about to commit the anti-pattern you guard against.

04 The Long View

Don't port (Option B), with C as the fallback only if a real signal ever appears.

Three years out, the asset that compounds here is the skill's reputation as the tool that gives a solo dev exactly what they need and nothing more. The asset that depreciates fastest is bespoke structural vocabulary baked into day-one scaffolds. Lanes and Seq are a control system tuned for *this* workspace's load — multiple concurrent skill streams, a furnace trial, cross-agent ledgers. A weekend-app builder inherits the ceremony without the load that justifies it, and "every fat row is a rule violation" becomes a guilt mechanism for someone whose backlog is six lines. That's not adoption, that's friction the skill's name eventually carries.

The second-order effect the user is missing: scaffolds are a *distribution* decision, not a workspace decision. Every project born on the one-board model is a future migration liability the moment the model evolves again — and it will, because it's young here (the workspace adopted it "recently," the ritual's frontmatter doesn't even hardcode the lanes yet). Porting now hard-couples the installed base to a structure that hasn't stabilized. You'd be exporting v0.9 of your own thinking to projects you'll never touch again.

The soft seam is the tell. The honest note says today's shape and the graduation target only *softly* mismatch — generic `status`, not hardcoded lanes. That seam is doing real work: it lets the model prove itself at graduation, on real projects, before you commit the installed base. Don't weld it shut for elegance.

What compounds instead: let the one-board model earn its way in. Keep the simple default, let graduation carry the upgrade if and when bloat demands it. If in a year multiple scaffolded projects independently reinvent lanes, *that's* your signal — and you'll port a proven shape, not a fresh one.

05 The Veteran

Port a reduced form — Option C, specifically "lanes but no Seq, adopted at graduation, not day-one." Here's the pattern I've watched fail and the one exception that makes it work.

The failure: a tool team falls in love with the workflow that works *for them* and ships it as everyone's default. Kanban-for-everyone is the canonical version. It works in the meta-repo because the meta-repo has continuous multi-lane work, a retirement ritual someone actually runs, and an author who *likes* tending a board. A solo dev shipping a weekend app has none of that. For them, lanes + Seq + one-row-per-unit + thin-on-touch + retire-don't-re-accrete is five rules of ceremony guarding against bloat they will never generate. You'd be exporting your discipline as their tax. That's the "over-engineer the day-one experience" anti-pattern this project already named — and the board *is* the canonical instance of it.

Why this case isn't fully special: you told me the seam is *soft*. Graduation machinery already exists, the ticket frontmatter is already a generic `status` field, and bloat is already handled. So the problem the full port "solves" is mostly already solved. Option A buys you elegance in the meta-repo and a brownfield trap (D-013) in every scaffolded project that later disagrees.

The real exception: lanes are cheap and legible; Seq + roadmap-collapse + the two accretion rules are the expensive, opinionated part. So make graduation the moment the board appears — when a project has *earned* enough work to need lanes, the upgrade path lights up, citing its failure mode (multi-section sprawl at scale). Day-one stays simple. You ship the model you proved, but as an earned upgrade, not a birthright. That's the standard playbook honoring its own exception.
