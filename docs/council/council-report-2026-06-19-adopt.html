Council Report

design-system-adopt: build the skill, or ship a rule?

5 advisors · anonymous peer review · chairman synthesis

The question

Should Rex (single-developer, markdown-only Claude Code skill workspace) build a new design-system-adopt skill that ingests a finished 'Claude Design' artifact (a real ~1,200-line prop-driven React component library) and preserves it at three levels — tokens, components, AND composition/fidelity — accepting that it crosses his existing design-system-bootstrap skill's deliberate invariant ('never feature components')? Grounded failure from his real project the-council: the design-to-app handoff harvested ONLY token values; zero components adopted, every page rebuilt inline, composition drifted 100%. Sub-questions: (1) is 'port existing' safe where 'improvise' is not? (2) can composition fidelity be enforced by a skill/rule, or only a human gate? (3) is the fix a lightweight discipline + import rule rather than a heavy generator? (4) ROI for a single-dev workspace?

The recommendation

Council verdict

Don't build the heavy design-system-adopt skill — ship a ~20-line always-on import rule plus a present, cheap human diff-against-source gate, log the invariant-crossing as a decision regardless, and only revisit a generator after the rule survives three real handoffs.

Build the rule, not the skill — and don't pretend the rule is self-enforcing. A 300-line generator buys a second skill that overlaps design-system-bootstrap, contradicts its headline invariant, rots on every Claude Design export change, and adds root-pile sprawl your own CLAUDE.md legislates against — all to solve the cheap half while the expensive half (composition) stays exactly as unenforced as it was in the-council. Skip it. But the peer reviews exposed a flaw in the easy answer: a 'diff-against-source, no-merge' gate staffed by the one person who already failed to open the design is the same habit wearing a badge. So build the rule around the two distinctions the reviewers drew. FIRST, split composition into its two halves and enforce them differently: intra-app consistency is the half you CAN make mechanical even in a markdown workspace — a session-start check or emitted hook that greps for inline re-implementations of components that already exist in the imported library (the way your hooks already live-fire on blocked operations); that removes the cheap, high-frequency drift without a human. Fidelity-to-source genuinely needs a human — make that gate survivable by making it cheap and PRESENT, not diligent: the rendered reference pages live in the repo next to the surfaces and the import rule fires on every new-surface session. Presence beats discipline; the-council failed because the design was absent, not because Rex lacked willpower. SECOND, address staleness in the rule: the reference of record is re-snapshotted on each design change with a dated marker, and a surface diffed against a stale reference is flagged, not trusted. THIRD — non-negotiable per house rules — log the invariant crossing as a D-NNN decision either way; even the lightweight rule establishes a deliberate breach of 'never feature components,' and the decision entry is what stops future-Rex from reading the two skills as contradictory and deleting the wrong one. On ROI: a single-dev workspace is exactly where a maintained generator loses and a deletable rule wins. And the Stranger's challenge — is this even a real problem? — is answered by the rule itself: run it for three real handoffs. If fidelity holds, presence-plus-gate was the fix. If it drifts anyway, you've proven composition needs a human gate no skill can supply — and spent ~20 reversible lines learning it instead of 300 maintained ones.

The one thing to do first

On the NEXT real design-to-app handoff, before writing any rule or skill: drop the actual rendered Claude Design pages (screenshots + component source) into the repo as design/reference/, copy the real component files in, and add one ~20-line always-on import rule that cites its failure mode (100% composition drift, the-council) — 'a design library exists at design/reference/; import from it, never rebuild a surface it already covers; rebuilding requires a cited reason; diff each new surface against its dated reference page before commit.' Run that one cycle, then decide whether anything heavier earns its keep. Open the D-NNN decision entry recording the invariant crossing as part of the same change.

Where the council agrees

Near-total consensus on the diagnosis, which is rare and worth weighting. (1) Composition fidelity is the real problem and it is NOT in the library: even a perfectly ported component kit leaves layout, slotting, and page assembly to be invented fresh each surface — tokens-plus-components solves only the cheap half. (2) A markdown skill cannot mechanically enforce fidelity: it ships no build-time gate, no PR block, no forced reference-open; tokens were enforceable because compliance is mechanically checkable, fidelity is a judgment call. (3) The fix is wiring, not a generator: copy the real component files in, add an always-on import rule that cites its failure mode, lean on a human gate — reversible, cheap, house-style-compliant. (4) The root cause was ABSENCE, not a missing tool: the finished design wasn't in the build context at composition time, so nothing was looked at because nothing was there to look at.

Where the council clashes

Little genuine clash on the verdict — all five land at 'don't build the heavy skill, ship the rule.' The divergence is about whether even the rule will work. The Skeptic is most pessimistic: a rule is theater unless a mechanical enforcer exists. The Stranger attacks one rung lower: maybe there's no problem worth solving — one dev, one app, components possibly used once, '100%' with no count behind it. The Architect and Long View are constructive: presence + a gate probably works, and even if it doesn't, running it manually proves the variable cheaply. The clash is about confidence in the human gate, not about building the generator — nobody defends the generator.

Blind spots the council caught

The peer reviews surfaced four decision-relevant gaps the advisors missed. (1) Two different 'compositions' got conflated: intra-app consistency (does page 2 compose primitives like page 1?) is snapshot/lint-checkable WITHOUT a human; fidelity-to-source (does any page match the Claude Design?) needs a human. Different enforcers; the cheap mechanical win lives in the first, and every advisor blurred them together. (2) The gate's enforcer is the same habit that already failed: every recommendation relocates the unenforced discipline from 'copy the components' to 'pass the gate,' staffed by the same dev who skipped opening the design last time. (3) Re-export staleness: Claude Design regenerates on every design change, so any committed 'reference of record' silently goes stale. (4) The DECISIONS process: crossing the 'never feature components' invariant is precisely the costly, hard-to-reverse fork that CLAUDE.md says must be councilled AND logged — even the lightweight rule establishes 'adopt ports real components,' so it still needs a D-NNN entry.

Advisor responses

01 The Skeptic

Composition fidelity is unenforceable by a markdown skill, and that's the flaw that sinks the heavy version. In the-council the design existed, the components existed, and a human-plus-agent loop with a linter STILL rebuilt every surface inline. The drift wasn't caused by a missing skill — it was caused by the path of least resistance being 'write fresh JSX.' A design-system-adopt skill ships markdown; it cannot ship a build-time diff gate, block a PR, or force an agent to open the reference. So it says 'compose from these, diff against the design' — and the next agent rebuilds inline because it's faster and nobody's watching. You'd encode the unenforced axis a second time and call it enforcement. Sub-question 1 is a trap: 'port not invent' sounds safe, but JSX-to-app translation IS invention — routing, state wiring, data fetching, slot composition. The component boundary is where improvisation re-enters. Bootstrap's invariant was 'agents handed a design artifact improvise,' and adopt hands an agent a design artifact. Enforceable only if something mechanical checks it. Build that check first. If you can't name the mechanical enforcer, the skill is theater, and a 300-line generator is theater with a budget. Ship the rule, wire the gate, see if fidelity holds for one cycle.

02 The Architect

Composition fidelity is the entire problem, and tokens-plus-components doesn't touch it. Even if adopt ports TarotCard perfectly, an agent building a page still decides HOW to arrange TarotCard, Channel, and Waveform — invented fresh every time. You can port a component library and still get 100% composition drift, because composition lives in the page, not the library. So sub-question 1 is a trap: porting components is necessary and safe, but it solves the cheap half. The expensive half — layout/structure — has no artifact to port unless the Claude Design system ALSO shipped the assembled pages, not just the component kit. Reframe: the variable you're optimizing isn't 'should the skill write components.' It's 'what is the reference of record, and is it present at the moment of composition.' The drift happened because the finished design was ABSENT from the build context. Nothing was looked at because nothing was there to look at. Simplest first move that proves it: skip the skill. Next handoff, drop the actual rendered Claude Design pages (screenshots + component source) into the repo as design/reference/, add one rule — 'every new surface diffs against its reference page before commit; no reference, no merge.' Run it once. If fidelity holds, your fix was presence + a gate, not a generator. If it drifts anyway, you've proven composition needs a human review gate, and no skill saves you. Either outcome kills a wrong variable cheaply.

03 The Stranger

Composition fidelity drifted '100% of the time' — but I can't see a single number behind that. How many surfaces? Three pages or thirty? 'Citable failure' is doing a lot of work for one project observed once. A failure you can name in a quotable sentence is not the same as a failure you've measured. Who is actually affected? One developer, on his own projects. There is no team, no handoff to a stranger who never saw the design. The 'we' in 'we enforced token compliance' is one person. So the real question isn't 'can a skill enforce fidelity' — it's 'why didn't Rex, the only person in the loop, open the design and copy the components when he wrote the app?' If the answer is 'I forgot / it was easier to retype,' a skill won't fix a habit; it'll be another file he forgets to invoke. Unspoken assumption: that the components are worth preserving as-built. TarotCard, Waveform, Channel sound bespoke to one app. 'Reusable library' describes their form, not their reuse. Have any been reused across two projects? If not, 'adopt and preserve fidelity' is preserving a thing used once. Build the thin discipline-rule — reversible, tests the habit; if Rex still rebuilds inline anyway, you've learned the generator would've been forgotten too, cheaply.

04 The Long View

Composition divergence didn't drift because nobody built the right tool — it drifted because three years of incentives all point at 'rebuild inline.' An adopt skill ships once, but the gravity that beat it last time keeps pulling every session. Inline JSX is the local minimum — faster to type, no import to reconcile, no design file to re-open. A markdown skill cannot out-pull that gravity at build time; only a gate the agent must pass through can. Play it forward: build the heavy generator and in three years you own a second skill that overlaps bootstrap, contradicts its headline invariant, and depreciates every time Claude Design changes its export shape — future-Rex pays the tax of remembering which, the root-pile sprawl your CLAUDE.md legislates against. Generators rot; the ported library rots faster. What compounds instead is the rule and the failure citation — 'when a Claude Design system ships real components, adopt the files and compose from them — never rebuild; the unenforced axis drifts every time' — which survives export-format churn and ports to the next design tool. Pick option 3: thin wiring plus an always-on fidelity rule, teeth being a human diff-against-design review gate. Don't build the generator until you've run the rule manually three times. A rule you delete costs nothing; a skill you maintain costs forever.

05 The Veteran

Composition fidelity drifted because nobody was forced to look at the design — and a skill cannot force that. The pattern I've watched a dozen times: teams enforce the cheap-to-check axis (tokens, a linter) and declare the expensive-to-check axis 'covered' because a tool now exists. Building adopt risks repeating the exact failure one level up — generated component files satisfy 'components were ported,' composition drifts again, except now you trust it less because a skill blessed it. The standard playbook is right here and your case is not special: the fix for 'components got rebuilt instead of imported' is wiring, not a generator. Copy the component files in, add an always-on rule — 'a design library exists at X/; import from it, never rebuild a surface it already covers; rebuilding requires a cited reason.' That rule cites its failure mode (100% composition drift, the-council), exactly your house style; ~20 lines, reversible, ports the real components rather than a generator's idea of them. Be honest on enforcement: tokens were enforceable because compliance is mechanically checkable; fidelity is a judgment call, so it lands on a human review gate, not a skill. ROI: single-dev workspace, reversible problem, real fix is a rule. Don't build the heavy skill; ship discipline + import rule, watch two real handoffs, then ask whether a generator earns its keep.
